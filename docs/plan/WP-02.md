# WP-02 — Protocol clients

Maps to roadmap **M2**. Crate: `vyane-protocol`.

## Goal

Speak the HTTP model protocols. Implement `vyane_core::ChatClient` for OpenAI
Chat Completions and Anthropic Messages (full non-streaming **and** SSE
streaming, normalized to `StreamEvent`), plus OpenAI Responses (types +
non-streaming; streaming stub returns `Unsupported`). Map every failure onto
`ErrorKind` faithfully.

## Scope (modules)

- `openai_chat` — `ChatClient` for OpenAI Chat Completions: `complete` + `stream`.
- `anthropic_messages` — `ChatClient` for Anthropic Messages: `complete` +
  `stream`.
- `openai_responses` — `ChatClient` for OpenAI Responses: `complete` only;
  `stream` left as the default trait impl (returns `Unsupported`).
- `retry` — exponential backoff + jitter, shared by all clients.
- `wire` — request/response serde types per protocol; `ChatRequest` →
  wire-format translation (system-message placement differs per protocol) and
  wire response → `ChatOutcome`.

HTTP via `reqwest` (workspace features already include `json`, `stream`,
`rustls-tls`, `gzip`). SSE parsing is hand-rolled over the byte stream — no new
dependency.

## Behaviour

- **Auth styles.** `AuthStyle::Bearer` ⇒ `Authorization: Bearer <key>`;
  `AuthStyle::XApiKey` ⇒ `x-api-key: <key>` (plus the Anthropic version header).
  Credential comes from the `Endpoint.auth` `Secret`.
- **Streaming normalization.** SSE frames map to `StreamEvent`:
  `Delta(String)` for answer text, `ReasoningDelta(String)` for thinking,
  `Usage(Usage)` near the end, `Done { finish_reason }` at clean close.
  **Liveness comes from `Delta`/`Done`, never from reasoning traffic** — some
  relays emit zero reasoning deltas even when the model reasons; a client that
  waits on reasoning to prove liveness is wrong.
- **Retry.** Exponential backoff + jitter on `429` / `5xx` / transport errors;
  configurable max attempts; honor `Retry-After` when present. Non-retryable
  statuses (e.g. `401`/`403`) fail immediately.
- **Error mapping** onto `vyane_core::ErrorKind` (this classification drives
  kernel failover, so it must be exact):
  - `401` / `403` → `Auth`
  - `429` → `RateLimited`
  - `5xx` / malformed body / refused request → `Protocol`
  - DNS / connect / TLS / broken stream → `Transport`
  - requested capability absent (Responses streaming) → `Unsupported`
- **Params mapping.** `GenParams.max_output_tokens` maps to each protocol's
  output-limit field; `temperature`, `top_p`, `effort` and `extra` mapped where
  the protocol supports them (`extra` applied last). Leaving `max_output_tokens`
  unset must send no limit.
- **Response parsing.** Populate `ChatOutcome { text, usage, model_echo,
  finish_reason }`; `model_echo` is the server-echoed model id (used to verify a
  relay actually served the requested model). Parse `Usage { input_tokens,
  output_tokens, reasoning_tokens?, cached_input_tokens? }` when reported.

## Frozen interfaces it implements

- `#[async_trait] ChatClient`: `fn protocol(&self) -> Protocol`,
  `async fn complete(&self, req: ChatRequest) -> Result<ChatOutcome>`,
  `async fn stream(&self, req: ChatRequest) -> Result<BoxStream<'static,
  Result<StreamEvent>>>`.
- Consumes `ChatRequest { model: ModelId, messages: Vec<ChatMessage>, params:
  GenParams }`, `ChatMessage`/`Role`.
- Produces `ChatOutcome`, `StreamEvent::{Delta, ReasoningDelta, Usage, Done}`,
  `Usage`.
- `Endpoint` / `AuthMaterial` / `AuthStyle` for wiring auth; errors via
  `VyaneError` / `ErrorKind` / `Result`.

## Acceptance (mechanically checkable)

All against `wiremock` (workspace dev-dep) — **no live network**:

- **Success:** `complete` returns the expected `text`, parsed `Usage`, and
  `model_echo` for each of the three protocols.
- **401:** maps to `ErrorKind::Auth`, and the client does **not** retry.
- **429 with `Retry-After`:** retried, honoring the header, up to the configured
  max attempts, then surfaces `ErrorKind::RateLimited`.
- **5xx:** retried, then surfaces `ErrorKind::Protocol`.
- **Stream chunking (split frame):** an SSE payload split across transport
  chunks mid-frame is reassembled into the correct ordered `StreamEvent`s and a
  terminal `Done`.
- **Malformed JSON:** garbage / truncated body yields `ErrorKind::Protocol`
  (never a panic).
- **Responses streaming stub:** `openai_responses.stream(..)` returns
  `ErrorKind::Unsupported`.
- **No panics on garbage:** fuzzy/garbage inputs across all parsers return
  `Err`, never panic.
- **Clippy clean** under `-D warnings`; no `unwrap` in non-test code.

## Non-goals

- No provider registry / config parsing (WP-01 supplies the `Endpoint`).
- No harness/subprocess behaviour (WP-03).
- No caller-facing retry policy beyond transport-level backoff (kernel owns
  cross-target failover).
- Full OpenAI Responses streaming is deferred; the stub must be explicit.

## Notes

- Keep wire types private to the crate; only the `vyane-core` types cross the
  boundary.
- Jitter must be bounded and the backoff base configurable so tests are
  deterministic (inject the sleeper / clock).
- Do not log request or response bodies at info level — they may contain prompt
  text; the ledger stores digests for a reason.
