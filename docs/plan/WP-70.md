# WP-70 · Bounded manual goal pursuit

Status: in progress

## Goal

Add the production manual outer loop that repeatedly verifies one in-progress
goal, dispatches one fresh runtime segment, and reverifies until acceptance is
achieved or an explicit budget/pause condition stops further work.

## Delivered slice

- `vyane-goal::GoalPursuer` owns the bounded verify → fresh segment → reverify
  loop behind an injected async `GoalSegmentRuntime`.
- Every verification keeps the WP-69 immutable artifact and fenced criterion
  mutations. Only the pursuer may call `done`, and only after all acceptance
  results are satisfied.
- Missing criteria and final manual confirmation pause without running a model.
  Segment count, consecutive verifier/runtime failures, per-segment timeout and
  overall timeout are explicit, validated bounds. Runtime cancellation pauses
  immediately without starting another verify/segment round.
- Every segment is sessionless and receives a bounded timeout no longer than
  the remaining overall budget. Runtime output is not copied into goal truth;
  progress keeps only bounded stage/status and an allowlisted run identity.
- `vyane goal pursue` requires the exact active lease holder, renews that lease
  against the remaining pursuit budget, and production-wires each segment through the ordinary Vyane
  dispatch/failover kernel with an explicit target, canonical workdir and
  sandbox. Until authenticated goal service wiring exists, this production CLI
  surface is limited to the same `local` single-user owner used by dispatch.
  Target/config and goal status/lease preflight before any verifier or model effect.
- External lifecycle changes stop the loop rather than being overwritten.

## Non-goals

- This is explicitly invoked manual pursuit, not daemon auto-pursuit or restart
  replay. Counters are returned and progress is durable, but there is no second
  writable pursuit snapshot.
- No quota/continuity handoff, approval/kernel takeover, authenticated goal
  service, native-session reuse, network/board verifier or implicit resume.
- A queued/paused/unleased goal is not silently started or resumed; the caller
  must claim it and pass the exact `--worker` first.

## Verification gate

- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`
- fake-runtime domain tests for achieve/manual/no-criteria/segment/failure/lease
- fake CLI-harness acceptance proving production dispatch, reverify and done
