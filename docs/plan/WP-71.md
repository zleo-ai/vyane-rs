# WP-71 · Lease-fenced pursuit checkpoints

## Goal

Make explicit manual goal pursuit restartable without weakening the owner,
worker, lifecycle, or lease boundaries established by WP-60 through WP-70.

## Delivered

- Goal schema v4 adds one mutable `(owner, goal_id)` pursuit checkpoint. The
  typed JSON payload is bound to a SHA-256 digest and redundant relational
  envelope fields are checked on every read.
- Every checkpoint write uses checkpoint-revision CAS plus the current goal
  revision and claim generation. It also appends the corresponding progress
  event and updates the goal snapshot in the same SQLite transaction.
- The checkpoint retains the canonical workdir, runtime label, original start
  time, lifetime segment/failure counters, last allowlisted run id, last
  verification-artifact id, and `running`/`paused`/`achieved` status.
- `GoalPursuer` records or adopts the checkpoint at start, verification,
  segment start/completion/failure, pause, and achievement. A new lease holder
  can continue a paused checkpoint after database reopen; stale workers and
  stale generations fail closed.
- Resumed or manually started `in_progress` goals with no lease can establish a
  fresh claim. Expired tenures still require explicit `reclaim`, preserving the
  distinction between unleased work and abandoned leased work.
- `goal get --json` returns the checkpoint beside immutable verification
  artifacts. Segment and failure limits remain lifetime counters across
  restarts; the overall wall-clock timeout remains a budget for the current
  invocation.

## Explicit limits

- No daemon automatically selects, claims, resumes, or pursues goals.
- No quota/continuity handoff, approval/kernel takeover, authenticated goal
  service/API, or network/board verifier is added here.
- Runtime-native sessions are not resumed. Every WP-70 work segment remains a
  fresh sessionless dispatch.
- External lifecycle or lease drift stops the old pursuer without a final
  checkpoint write because it no longer owns mutation authority.

## Verification

- Store contract coverage proves revision/generation/worker fencing,
  transactional progress, digest/envelope validation, migration, and adoption.
- Pursuit coverage closes and reopens the database, resumes and claims with a
  different worker, continues at the next segment index, preserves the
  original start time, and consumes the remaining lifetime budget.
- Achieved checkpoints retain terminal counters plus the last run and
  verification identifiers.
- Mutation check: forcing `segments_started` to zero during checkpoint adoption
  makes `paused_checkpoint_survives_reopen_and_new_lease_continues_lifetime_budget`
  fail at `tests/pursuit.rs:241` because completed segments then exceed started
  segments. Restoring lifetime-counter adoption returns the test to green.

CON-07 remains `partial`: restartable manual state is now present, but daemon
automatic pursuit and the other limits above remain separate work.
