# WP-01 — Config & Provider

Maps to roadmap **M1**. Crates: `vyane-config`, `vyane-provider`.

## Goal

Turn TOML configuration into resolved, runnable targets. Given a profile name
(or a `provider/model` pair, or a failover chain), produce a fully-resolved
`vyane_core::BoundTarget` — and, for harness targets, the `EnvPolicy` inject set
that gives the child a self-contained environment. Configuration never stores
key material.

## Scope

- **`vyane-provider`**
  - `Provider` registry: for each `[providers.<id>]`, hold `base_url`,
    `api_key_env`, `auth_style`, `protocol`, `default_model`, and `extra`.
  - Resolve a provider + explicit/`default` model into a `vyane_core::Endpoint`:
    read the secret from the env var named by `api_key_env`, wrap it as
    `AuthMaterial { style, secret }` (`Secret` from `vyane-core`).
  - Per-provider env-injection rules → produce the `EnvPolicy.inject` map for
    harness runs (endpoint/key/model → the env vars each CLI expects).
- **`vyane-config`**
  - Parse the TOML model (below) with `serde` + `toml`.
  - Layering with precedence; profile resolution to `BoundTarget`; failover
    chain resolution (each element independent and fully resolved).
  - Optional gitignored secrets file (`KEY=VALUE` lines) folded into env lookup.

### TOML model

- `[providers.<id>]`: `base_url`, `api_key_env` (**indirection only — the config
  never stores the key itself**), `auth_style` (`bearer` | `x_api_key`),
  `protocol` (`openai_chat` | `openai_responses` | `anthropic_messages`),
  `default_model`, `extra` (free table, e.g. `max_output_tokens`).
- `[profiles.<name>]`: `provider` + `protocol` + `harness` + `model` + params
  bundle, resolving to one `vyane_core::BoundTarget`. `harness` is optional
  (`"none"` / omitted ⇒ `DirectHttp`, `Option<HarnessKind>::None`; otherwise
  `CliWrap`). Optional `sandbox` (`read-only` | `write` | `full`).
- `[profiles.<name>].failover = ["profileA", "provider/model", …]`: an ordered
  list; **each element independently and fully resolved.** A model id is never
  carried across providers — an element either names another profile or pins its
  own `provider/model` pair.

### Layering precedence (lowest → highest)

1. built-in defaults
2. user file `~/.config/vyane/config.toml`
3. project file `.vyane/config.toml`
4. explicit CLI args

Higher layers override lower **per field** (a project file overriding one
provider field must not drop the others). Use `dirs` for the config home.

## Frozen interfaces it implements / produces

From `vyane_core`:

- Produces `BoundTarget { target: Target, transport: AdapterTransport,
  endpoint: Option<Endpoint>, params: GenParams }`.
- `Target { provider: ProviderId, protocol: Protocol, harness:
  Option<HarnessKind>, model: ModelId }`.
- `Endpoint { base_url, auth: Option<AuthMaterial> }`,
  `AuthMaterial { style: AuthStyle, secret: Secret }`, `AuthStyle::{Bearer,
  XApiKey}`.
- `AdapterTransport::{CliWrap, DirectHttp}`, `Sandbox::{ReadOnly, Write, Full}`,
  `Protocol::{OpenaiChat, OpenaiResponses, AnthropicMessages}`,
  `HarnessKind::{ClaudeCode, CodexCli, OpenCode, Other}`.
- `GenParams { temperature, top_p, max_output_tokens, effort, extra }`,
  `Effort::{Low, Medium, High, Xhigh}`.
- `EnvPolicy` (produce the `inject` set; `mode` defaults to `Scrub`).

The config layer resolves; it does not execute. It touches no trait objects.

## Acceptance (mechanically checkable)

- **Precedence unit tests:** built-in < user < project < CLI, verified per
  field (overriding one field of a provider leaves its siblings intact).
- **Profile resolution:** a `harness = "none"` profile resolves to
  `AdapterTransport::DirectHttp` with `Target.harness == None`; a
  `harness = "claude-code"` profile resolves to `CliWrap` with
  `Some(HarnessKind::ClaudeCode)`.
- **Failover resolution:** a chain of mixed profile-names and `provider/model`
  pairs resolves to `Vec<BoundTarget>` in order, where each element's
  `Target.model` is paired only with its own `Target.provider` (a model-leak
  assertion).
- **Missing-env error names the variable:** when `api_key_env = "FOO_KEY"` is
  unset, resolution fails with `ErrorKind::Config` and a message containing the
  exact string `FOO_KEY`.
- **Round-trip tests:** parse → in-memory model → serialize → parse yields an
  equal model for providers and profiles.
- **`profiles.example.toml` stays in sync:** a test parses the repo's
  `profiles.example.toml` and asserts it resolves without error, so the example
  can never drift from the parser.
- **Secrets never serialize:** a compile-or-runtime test that a resolved
  `Endpoint`/`AuthMaterial` cannot be `serde`-serialized (the `Secret` type
  forbids it), guarding the "no key material persisted" invariant.

## Non-goals

- No HTTP calls, no process spawning (that is WP-02 / WP-03).
- No routing policy or model selection heuristics (that is `vyane-router`, v0.3).
- No live config reload / file watching.

## Notes

- `api_key_env` is indirection **only**; the key value lives in the process
  environment or the gitignored secrets file, never in tracked config.
- The secrets file and any real `config.toml` are gitignored; only
  `profiles.example.toml` is tracked (see `.gitignore`).
- Keep the example config the single source of truth for the config shape —
  document new fields there first.
