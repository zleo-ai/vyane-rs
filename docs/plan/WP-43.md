# WP-43 — Bounded dark native turn driver

## Goal

Add a provider-neutral model-to-tool-to-model driver that can exercise only the
already-authorized model and registry boundaries, while remaining unavailable
to production assembly until native execution authority is complete.

Integrated into the current clean public baseline.

## Bounded serial contract

`vyane-harness::native::NativeTurnDriver` owns an
`Arc<dyn AuthorizedToolChatClient>`, one immutable `ToolRegistry`, one
`PermissionPolicy`, and validated `NativeTurnLimits`. The default is eight
logical model turns, zero is invalid, and 32 is the hard ceiling. One response
may request at most one tool call, so each run is strictly
model-send → optional-tool → model-send rather than parallel tool fan-out.

The initial request must be valid and its advertised tool-name set must exactly
equal the executable registry-name set before the first model send. Tool
descriptions and JSON schemas are non-authoritative guidance for the model:
they are not execution authority and cannot constrain arbitrary model output.
Every `NativeTool` must validate the actual arguments it receives and enforce
its own semantic and capability boundary.

## Validation before effects

- The initial request, every subsequent request, and every model response pass
  the bounded typed tool-chat validator.
- Before permission evaluation or a tool future can be polled, the driver adds
  the exact assistant turn plus a worst-case bounded tool result to a cloned
  request and validates the complete next transcript. Duplicate call ids,
  unresolved pairing, message limits and envelope exhaustion therefore stop
  before the tool boundary.
- Invalid JSON arguments never become executable arguments. They produce one
  static error tool result that does not echo the invalid source text.
- Denied, invalid and unknown calls, plus bounded tool errors, remain
  model-facing results and may continue only within the same bounded run.

## Live authority and non-replayable stops

Every model request uses `AuthorizedToolChatClient`; every known allowed tool
uses `ToolRegistry::execute_authorized`. Logical turn and tool ordinal are
1-based, and the only possible ordinal is one. The underlying protocol client
continues to own 1-based physical wire attempts and revalidates each attempt.

The driver marks `tool_side_effects_possible` before entering an allowed,
known tool boundary. An initial transcript that already contains a tool call or
result is also treated conservatively. Before any possible tool activity,
ordinary model errors keep their original classification; cancellation and
timeout become typed stops. After tool activity may have crossed its boundary,
a model, authority or transcript failure becomes
`AbortedAfterToolActivity { kind }`, never an outer failover-eligible error.

Other typed terminal stops are:

- `Completed` and `Refused`;
- `ApprovalRequired`, with the exact bound approval plan available to an
  explicit caller but no automatic replay;
- `UnsupportedParallelCalls` and `ToolChoiceViolation`;
- `Cancelled`, `TimedOut`, and `BudgetExhausted`.

Refusal wins before permission or tool execution, including when the provider
response also carries a tool call. `ToolChoice::None`, `Required`, and
`Named` are enforced at every request boundary.

## Bounded result surface

Usage from validated responses is accumulated with saturating arithmetic for
input, output, reasoning and cached-input tokens. A completed/refused stop
retains the final assistant reply but not reasoning, tool calls, transcript or
tool output. An approval stop necessarily retains its exact bound
`ApprovalPlan` for an explicit caller. Across variants, custom `Debug` redacts
final reply and approval contents, provider error text and tool data; the
outcome has no Serde implementation.

## Acceptance

- Contract tests cover zero/default/hard limits, one-based serial effects,
  exact advertised/registry name sets, saturating usage and redacted output.
- Boundary tests cover invalid JSON without source echo, refusal, parallel
  calls, all tool-choice variants, ask without replay, cancellation during a
  hanging tool, timeout before tool polling, and budget exhaustion without an
  extra send.
- Preflight tests reject duplicate call ids and a full message budget before a
  second tool poll. Authority tests prove revocation prevents the next physical
  send or tool poll, while post-tool provider errors become typed non-failover
  stops.
- Repository format, Clippy-with-warnings-denied, workspace tests, independent
  review and staged secret scanning remain merge requirements.

## Integration state and non-goals

This driver is a dark library seam, not a `Harness`. No factory, service
operation, CLI, daemon or resident runtime constructs it, and ordinary dispatch
is unchanged. It does not provide:

- trusted read/write/bash/search tools or host-enforced sandboxing;
- session-bearing authority, `NativeSessionDomain` revalidation or native
  resume;
- checkpoint prepare/publish or revision-fenced session commit;
- approval persistence, one-shot consumption or approval resume;
- resident AgentRun supervision, controller recovery or automatic replay.

Production assembly must explicitly combine the driver with a live authority,
trusted tools whose internal side effects revalidate independently, and the
remaining session/checkpoint/approval contracts before any `NativeHarness` is
registered.
