# WP-62 — authenticated MCP workflow control

## Goal

Allow an MCP client to submit, inspect, and idempotently cancel one durable
resident-daemon workflow without moving daemon discovery, credentials, or task
storage into `vyane-mcp`.

## Contract

- `vyane-mcp` defines an object-safe `WorkflowControl` port and closed public
  request/view/error types.
- The base constructor remains a six-tool library surface. An embedding process
  may inject the port; `vyane mcp` injects the production adapter and exposes
  nine tools.
- Submission accepts a caller-owned canonical UUIDv7, a bounded self-contained
  workflow source bundle, and bounded variables. Exact-payload retry semantics
  remain owned by the daemon.
- The MCP caller cannot provide an owner, controller, lease, token, daemon
  address, or execution directory. The CLI freezes its startup directory and
  re-authenticates the verified resident daemon for every operation.
- Public results contain only caller id, lifecycle state, and an optional closed
  failure code. Transport and daemon messages never cross the MCP boundary.
- An indeterminate submission is reported as `outcome_unknown`; callers must
  inspect the same UUIDv7 before retrying the identical request.

## Acceptance

- Fake control tests drive the real rmcp protocol and prove the default six-tool
  surface, injected nine-tool surface, strict schemas, bounds, redaction, and
  idempotent cancellation contract.
- A CLI acceptance test drives a real MCP child through an authenticated real
  resident daemon for submit, status, and repeated terminal cancellation.
- Repository fmt, clippy with warnings denied, and workspace tests pass.

## Remaining gap

This package narrows `INT-03`; it does not implement a unified task tool,
board/collaboration tools, distinct-principal owner context, or full reference
MCP parity.
