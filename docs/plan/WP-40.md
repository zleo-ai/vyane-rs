# WP-40 — Resident workflow daemon and authenticated local control

## Goal

Add one resident Vyane supervisor that can own workflow execution after the
submitting terminal exits. The daemon reuses WP-39's metadata-only task domain,
keeps workflow payloads out of SQLite, and exposes an authenticated loopback
control API for submit, status, and cancellation.

This work does not turn the task database into a replay queue. A daemon restart
first cleans exact child controllers and marks abandoned workflow tasks
`interrupted`; it never silently re-executes a payload from an earlier process.

## Command surface

- `vyane daemon run [--addr 127.0.0.1:9722]` runs the supervisor in the
  foreground. Non-loopback addresses are rejected.
- `vyane daemon start [--addr ...]` starts the same command in a new session,
  writes owner-only logs/control files, waits for authenticated readiness, and
  then exits.
- `vyane daemon status [--json]` requires both an exact daemon process identity
  and a successful authenticated health response.
- `vyane daemon stop` revalidates the exact process identity before every TERM
  or KILL and waits for graceful shutdown before escalating.
- `vyane workflow submit <file> [--var k=v] [--id UUIDV7] [--json]` submits to
  the resident daemon and never falls back to foreground execution when the
  daemon is absent. `--id` is the reconciliation/idempotent-retry key.
- `vyane workflow status <id> [--json]` reads the daemon-owned workflow task.
- `vyane workflow cancel <id> [--json]` durably requests cancellation before
  signalling the matching in-memory token.
- Existing `workflow run`, `resume`, and `list` remain explicit foreground and
  journal-oriented commands.

## Local protocol and trust boundary

- The listener is loopback-only and has no permissive CORS layer.
- Every endpoint, including health, requires a per-start 256-bit bearer token.
  The token is generated from the operating system CSPRNG and stored only in an
  owner-readable file below the Vyane data directory.
- The control descriptor is mode `0600` and contains only schema version,
  loopback address, daemon instance id, PID/PGID, start time, and Linux birth
  fingerprint. The token is kept in a separate mode-`0600` file; the data
  directory is mode `0700`.
- Request bodies have a fixed upper bound. Error responses are bounded and do
  not echo workflow payloads, provider errors, credentials, or raw model output.
- This is protection from accidental cross-process/browser access, not a
  same-UID sandbox. A hostile process running as the Vyane user can read the
  user's private files; stronger isolation requires a separate OS identity,
  container, or VM.

## Workflow identity and source bundle

- `WorkflowRunId` is a validated newtype. Only canonical lowercase, hyphenated
  UUIDv7 strings are accepted by constructors, serde, journal reads, CLI
  arguments, URL parameters, and filesystem path construction.
- The client generates the id by default after it has collected the source
  bundle and canonicalized the submission directory. It writes and flushes the
  id to stderr before POST; `--json` uses a one-line structured stderr event so
  successful stdout remains a pure task view. `--id` accepts the same canonical
  newtype for reconciliation or an exact retry.
- The workflow task id and journal run id are exactly the client-generated
  value. The daemon supplies it to the engine after the task row is durably
  created.
- The client reads the workflow file and any declared `prompt_file` values. It
  submits a bounded source bundle; the daemon never resolves a client-supplied
  path against its own filesystem.
- Bundle paths must be normalized relative paths with no absolute root, parent
  traversal, current-directory component, backslash, NUL, symlink escape,
  duplicates, or undeclared extra entries. The server requires every referenced
  prompt and rejects missing content and non-regular files.
- Semantic limits are 1 MiB for the TOML, 4 MiB for each prompt, 16 MiB total,
  128 prompt entries, and 4,096 bytes per portable bundle path.
- The normalized v1 workflow hash is domain-separated and length-framed over
  the TOML and every resolved prompt entry in deterministic path order. An
  exact legacy journal hash is accepted once and atomically migrated; new
  journals always use v1. The bundle lives only in the authenticated request
  and the daemon's live execution state. SQLite remains metadata-only.

## Submission identity and execution context

- The request carries the client's canonical absolute UTF-8 current directory,
  bounded to 4,096 bytes. For a new submission it must still exist and be that
  exact canonical directory. A missing step `workdir` becomes this directory,
  a relative `workdir` is joined to it, and an absolute `workdir` is preserved.
- Variables are bounded to 128 entries, 256 bytes per key, 1 MiB per value, and
  4 MiB in total. Empty/NUL keys and NUL values are rejected; validation errors
  identify the entry but do not echo its value.
- A domain-separated submission digest covers the v1 source hash, execution
  directory, and variables in sorted key order. An existing id is idempotent
  only for the exact `(owner=local, kind=Workflow, origin=Daemon)` scope,
  `target_key=workflow`, and identical digest. Any mismatch returns one generic
  conflict and never replays the old request.
- Exact retries return the existing task before validating current target
  configuration, so configuration drift cannot hide a previously accepted
  submission. Concurrent identical creates have one durable creator; the other
  caller observes that same task.

## Durable metadata and journal ownership

- Daemon workflows use `TaskKind::Workflow` and a distinct daemon origin.
  REST recovery/cancellation must match `(RestAsync, Dispatch)` and daemon
  recovery/cancellation must match the workflow pair; neither frontend may
  claim the other's rows.
- Submission runs inside a tracked, cancellable initializer. For a new id the
  order is: validate bundle/variables/execution context and route targets,
  create durable `queued`, register initialization cancellation, attach the
  daemon controller plus a 30-second lease, write the initial journal using the
  same run id, publish the exact live token/watcher, and finally release the
  workflow start gate.
- Lease renewal runs every 10 seconds and always compares revision, executor
  epoch, owner, and current unexpired lease. A stale worker cannot renew or
  settle a task claimed by a newer daemon.
- Each workflow transition updates its existing JSON journal atomically. The
  journal directory is mode `0700`; temp and final files are mode `0600` and
  use create-new unique names plus rename. Journal path construction never
  accepts an unvalidated string.
- The task database is authoritative for control status. Journal state can lag
  after a crash and is presented as execution detail, never as permission to
  replay or signal a process.

## Harness controller collection

- A daemon workflow can run multiple harness targets concurrently, so its
  durable controller is a set rather than WP-39's single detached sidecar.
- Every harness `Started` synchronously records an owner-only controller entry
  containing exact PID/PGID/start/birth identity before the real CLI start gate
  is released. A failed publication kills and reaps the fresh harness.
- `Stopped` removes only the exact matching birth after the original process
  group is known empty. Different pending identities are never overwritten.
- Reporter updates use a bounded advisory lock. Controller entries contain no
  prompts, arguments, environment, result, or error text.
- Explicit cancellation and startup recovery verify every exact controller,
  signal only while its sentinel identity is still live, and fail closed on a
  dead sentinel plus a live numeric PGID. Controllers that deliberately leave
  the sentinel process group are outside the PGID containment guarantee.

## Daemon lifecycle and recovery

- `serve` and `daemon run` share the per-data-directory supervisor lock. Bind
  succeeds before recovery mutates metadata; the lock remains held until all
  tracked futures have stopped or been aborted and awaited.
- On startup the daemon enumerates every active daemon workflow page. It first
  attempts best-effort cleanup of every exact nested controller, retaining
  fail-closed entries it cannot safely resolve, then marks the row `interrupted`
  with `WorkerLost`. A malformed controller entry does not prevent other
  entries or task metadata from being handled. It does not resume the workflow
  journal.
- Admission and initialization have an explicit shutdown gate. Client
  disconnect cannot drop the future between durable create, attach, journal
  creation, and tracked spawn.
- Signal handling closes workflow admission before waiting on HTTP. The daemon
  gives admitted HTTP requests up to 10 seconds, then requests durable
  cancellation, signals exact tokens, drains initializers/workers/controller
  cleanup/watchers, and performs exact interruption passes before releasing the
  lock. Cooperative HTTP and workflow phase budgets total 68 seconds;
  `daemon stop` revalidates identity and escalates from TERM after 75 seconds.
  A safety wait for an already-running blocking SQLite call may outlive an
  internal phase budget rather than releasing ownership early.
- Descriptor/token publication occurs only after listener bind, supervisor
  lock, database open, recovery, router construction, and shutdown-handler
  installation succeed. Cleanup uses compare-and-remove so an old process
  cannot delete a newer daemon descriptor.

## Acceptance

- Canonical UUIDv7 validation rejects traversal, alternate UUID spellings,
  non-v7 UUIDs, and journal/request id mismatches.
- Journal directory/file permissions, unique-temp cleanup, atomic replacement,
  and concurrent-reader behavior are tested on Unix.
- A submitted workflow continues after the submitting CLI exits and reaches a
  terminal task row with the same id as its journal.
- Daemon absence or an invalid token fails closed; submit never executes the
  workflow locally as a fallback.
- Non-loopback bind, unauthenticated health, CORS preflight, oversized bodies,
  filesystem `prompt_file` resolution, and malformed bundles are rejected.
- Cancellation versus completion has one CAS winner; completion may remain
  authoritative after a cancellation race.
- A daemon killed mid-harness leaves exact controller entries. The next daemon
  cleans the verified groups, records `interrupted`, and does not replay.
- REST and daemon recovery tests prove that each frontend ignores both the
  other origin and the other task kind, and that scoped queries continue across
  multiple pages.
- A stopped or PID-reused daemon cannot be reported healthy or signalled from a
  stale descriptor. Start/status/stop retain no token or payload in argv,
  environment, SQLite, or process listings.
- Full workspace tests, Clippy with warnings denied, and the 13-crate publish
  preflight remain green.

## Delivery slices

1. Canonical workflow ids, caller-supplied engine ids, journal path/permission
   hardening, and source-bundle validation.
2. Shared supervisor primitives, daemon `run`, authenticated API, task/lease
   ownership, tracked execution, cancellation, and recovery.
3. Exact multi-controller workflow sidecars plus daemon `start/status/stop`.
4. Cross-frontend race tests, crash/restart acceptance, documentation, and
   publish verification.

## Non-goals

- Automatic payload replay or workflow resume after daemon restart.
- Pause/resume of a live daemon-owned task.
- Remote or multi-user daemon exposure.
- Moving prompts, workflow source, outputs, raw errors, logs, or journals into
  the SQLite task metadata store.
- Treating a process group as a security boundary against hostile same-UID
  code or a descendant that deliberately creates another session.
