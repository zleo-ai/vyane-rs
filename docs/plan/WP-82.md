# WP-82 · Separately authenticated bounded continuity runner

## Outcome

Consume the WP-79/WP-80 closed next-action projection for a finite goal set and
invoke only explicitly assembled queue or approved one-shot execution ports.
Read, queue and execute capabilities authenticate independently and resolve to
one frozen owner. Approval decisions remain exclusively manual.

## Contract

- `GoalContinuityRunnerAuthorityFactory` owns separate read, queue and execute
  authentication policies. Every configured purpose requires its own
  credential, and all authenticated contexts must resolve to the same owner.
  Opaque authorities contain no serializable credential or principal value.
- `GoalContinuityRunner` uses `GoalReadService` through the closed projection
  reader contract. It does not inspect request-supplied owner, target, workdir,
  sandbox, prompt or dispatch parameters.
- A queue port can be invoked only for `queue_approval`. An approved execution
  port can be invoked only for `execute_approval` with a non-empty durable
  approval ID. Both receive the exact goal revision, quota boundary, step ID,
  step kind and approval ID projected for that action.
- Every trusted port retains an opaque `OwnerContext`. Assembly compares that
  binding with the separately authenticated purpose context and rejects a
  missing, extra or differently owned port before any read or action.
- Port implementations must re-read and enforce that fence. Queue must use the
  existing idempotent approval queue transaction. Execution must atomically
  consume the exact approved one-shot approval before dispatch and settle the
  same approval afterward.
- The runner deliberately has no decision port. `decide_approval` and
  `manual_decision` always return `manual_decision_required`, even when queue
  and execution capabilities are present.
- `run_once` rejects empty, duplicate, malformed or oversized goal sets before
  any read. Goal count, concurrency and per-goal read/action timeout are hard
  bounded. A timed-out blocking store read keeps its concurrency permit until
  the underlying read really exits, so later goals cannot accumulate detached
  reads beyond the configured limit. Work is attempted once and reports remain
  in caller input order.
- Reports contain only a validated goal ID, closed projected action and closed
  status. They exclude owner, credentials, target, paths, prompt, approval
  record, output and raw errors.

## Verification gate

- Tests prove purpose credentials cannot substitute for each other and
  cross-owner capability or port assembly fails closed.
- Tests prove queue and execution receive exact projection fences, execution
  requires an approval ID, and a missing capability cannot fall back to a
  different port.
- Tests prove decision/manual actions never invoke either mutation port.
- Tests prove invalid sets cause zero reads and concurrency, timeout and stable
  report order are bounded, including a blocking read that outlives its caller
  timeout without releasing its permit early.
- Mutation tests must fail if purpose authentication, manual-decision routing or
  the execution approval-ID fence is removed.
- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`

## Explicit exclusions

- no HTTP, MCP or CLI runner endpoint;
- no concrete queue or execution adapter and no raw dispatch port;
- no automatic approve/reject decision;
- no observation ingestion authority, watcher or external fact inference;
- no retry, repair policy, periodic loop or daemon assembly;
- no goal completion or acceptance mutation;
- no runtime-native session resume;
- no reuse of the WP-80 REST bearer as queue or execution authority.

CON-07 remains `partial`. WP-83 may production-assemble fresh/sessionless native
trusted built-ins and a supported-platform host sandbox, but cannot weaken this
runner's decision or one-shot approval fences.
