# WP-77 · Explicit primary-resume approval and execution

## Outcome

Extend the existing durable continuity approval boundary to the ready
`resume_primary` step. A primary continuation can run only after an explicit
decision bound to the exact current plan, target, quota-reset signal and, when
required, the exact successful review and takeover chain.

## Contract

- `continuity-queue` accepts `resume_primary/resume_primary_after_reset` only
  when that exact approval-required step is visibly `ready`.
- A review-required resume binds the exact successful review approval and run.
  The review approval must still bind an intact successful takeover approval
  and run for the same goal and quota event.
- A policy that does not require review carries no review predecessor evidence;
  unexpected evidence fails closed.
- Queue and decision never dispatch. Execution still accepts only the durable
  approval ID, resolves exactly one target, revalidates provider, protocol,
  harness, model and canonical work directory, then atomically consumes the
  approval and marks the step `in_flight` before dispatch.
- The primary-resume prompt contains the approved target axes, quota-reset
  evidence and the available review/takeover approval and run evidence.
- Terminal dispatch evidence is persisted through the existing approval row and
  visible step transition. Success marks `resume_primary` done but does not
  complete the goal or satisfy acceptance criteria.
- The dispatch is a fresh, approved goal-continuation segment. It is not a
  claim of runtime-native session resume.

## Explicit exclusions

- no automatic approval, queueing, dispatch or retry
- no polling/continuity daemon integration
- no runtime-native session resume
- no authenticated HTTP/MCP goal service
- no review-check watcher or repair loop

CON-07 remains `partial` after this package.
