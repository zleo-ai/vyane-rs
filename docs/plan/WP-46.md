# WP-46 — Closed workflow effort routing and replay freeze

Status: delivered

Integrated into the current clean public baseline.

## Purpose

Close the repository-local explicit-effort gap for deferred workflow routing
without widening the trusted label surface or implying whole-workflow parity.
The contract must select one effective effort before dispatch, preserve it for
every failover attempt and replay surface, and fail closed where route hints do
not have unambiguous semantics.

## Delivered contract

`WorkflowRouteHints::effort` is a closed, protocol-neutral `Effort` value. TOML
accepts only `low`, `medium`, `high`, and `xhigh`; adapters remain responsible
for translating that value to their own wire representation.

Route hints are meaningful only for a deferred single target such as `auto`.
Validation rejects non-empty route hints on either:

- an explicit target that resolves during static validation; or
- `fan_out`, whose current shared `TaskSpec` cannot preserve independent
  per-target routing metadata.

These failures occur before workflow journal creation, task/process admission,
or network activity. Invalid typed effort is reported without echoing the
untrusted input.

The effective effort precedence is:

1. explicit workflow route effort;
2. configured effort of the selected profile;
3. the selected decision tier's default effort.

The resulting value replaces per-leg defaults across the complete admitted
failover chain and is recorded as canonical `routing.effort`. A user-supplied
`routing.effort` label is rejected as reserved routing metadata, while a generic
`effort` label is preserved only as ordinary user metadata and cannot influence
the canonical decision.

## Replay and idempotency

The admitted effective effort is part of the frozen route rather than a value
recomputed from mutable profile configuration. The same canonical value is
checked or reconstructed for:

- recorded automatic-route replay;
- detached worker target snapshots;
- daemon idempotency comparison;
- workflow source-bundle and prepared-journal resume validation.

Every reconstructed failover leg receives the frozen effort. A malformed or
missing recorded effort fails closed with a bounded diagnostic instead of
falling back to a current profile or tier default.

## Verification

Hermetic unit and acceptance coverage proves:

- workflow explicit effort overrides both selected-profile and tier defaults;
- selected-profile effort overrides a tier default when no workflow override
  exists;
- the tier default is used only when neither higher-precedence source exists;
- primary and failover wire requests use the same effective effort;
- reserved-label rejection and generic-label non-forgery;
- invalid values fail before journal or network activity without input echo;
- explicit-target and `fan_out` route hints fail before dispatch;
- frontier admission remains authoritative and cannot be bypassed by effort;
- recorded, detached, daemon-idempotent, and prepared-journal paths detect or
  preserve the frozen value.

The delivery gate is the repository's full `cargo fmt`, strict `cargo clippy`,
and workspace test suite, followed by independent review.

## Non-goals and remaining parity work

WP-46 does not change EXE-05 or EXE-06 from `different`. It does not provide:

- sanitized cross-implementation fixtures for automatic route precedence,
  frontier admission, or frozen replay;
- the shared typed `WorkflowPlan` or an optional compatibility frontend;
- a production-complete model-tier selection policy;
- history, benchmark, or feedback-based routing signals;
- automatic daemon restart replay or live pause/resume.

Consequently this package closes one concrete Rust implementation gap, not
workflow parity and not full Vyane parity.
