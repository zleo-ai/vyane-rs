# WP-55 — Live harness spawn authority

Status: delivered in the current clean public integration baseline.

## Outcome

`HarnessExecutionContext` can now carry a clone-safe, process-local
`HarnessSpawnAuthority` alongside the existing `PinnedWorkdir`. Claude Code and
Codex CLI capture and streaming paths revalidate it before every physical
wrapper spawn attempt and again immediately before releasing the real target.
Legacy harness implementations fail closed when handed a scoped authority they
do not implement.

The authority is deliberately executable runtime state. It is not serializable,
persisted, logged, or inherited by the child. Its callback must be trusted,
body-free on panic, and bounded; an indefinitely blocking callback can stall the
calling runtime and is outside this seam's failure-isolation guarantee.

## Start boundary

Authority-bearing execution requires a lifecycle reporter and therefore a
start-gated Unix sentinel. The sentinel is spawned before the target, reports
its exact process group, and waits for a private gate. Cancellation, the
absolute run deadline, and live authority are checked both before and after the
synchronous authority callback. A denial kills and reaps the still-gated group,
so the target cannot produce effects.

The sentinel itself starts with only a fixed `PATH`. Target variables, including
dynamic-loader variables, are rendered into a private inherited descriptor and
are sourced only after the gate is released. Empty target environments remain
empty. This prevents target startup configuration from executing inside the
trusted sentinel before authorization.

## Verification and limits

Tests cover both built-in CLI harnesses and both capture/streaming paths,
pre-spawn denial, revocation after `Started`, cancellation during authorization,
deadline expiry inside the final authorization callback, empty target
environments, loader-variable isolation, and combined pinned-workdir plus live
authority execution. The slice passed formatting, warnings-as-errors Clippy,
the `vyane-core` and `vyane-harness` test suites, and Rust 1.85 locked checking.

This is a dark prerequisite, not a Process AgentRun backend. No production
AgentRun component constructs the authority yet. Exact process identity,
owner-bound sidecar state, Process execution/recovery, concrete input/result
handback, resident production assembly, tree-cancel consumption, and public
AgentRun APIs remain open. The gated shell also cannot retry an `ETXTBSY` raised
by the target's later `exec`; only the physical wrapper spawn is retried.
