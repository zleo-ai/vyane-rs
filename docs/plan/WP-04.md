# WP-04 — Kernel

Maps to roadmap **M4**. Crate: `vyane-kernel`. **Depends only on `vyane-core`.**

## Goal

The orchestration state machine. `dispatch` runs a task against a chain of
targets with correct failover gating and a complete attempt trail; `broadcast`
fans the same task across N chains concurrently in input order. Every run
produces exactly one `RunRecord`, appended to the ledger and reflected in the
session store — on success **and** on failure.

## Scope (modules)

- `executor` — the executor abstraction and the factory seam.
- `dispatch` — single-task attempt loop, failover gating, `RunRecord` assembly.
- `broadcast` — concurrent dispatch over many chains, order-preserving.
- `digest` — prompt → `task_digest` (SHA-256, hex, first 16 chars).

The kernel names **no** concrete client/harness/ledger type. It holds them
behind `vyane-core` traits, injected by the assembler (the CLI layer).

### Executor seam

```
enum Executor {
    Chat(Arc<dyn ChatClient>),   // DirectHttp targets
    Agent(Arc<dyn Harness>),     // CliWrap targets
}

trait ExecutorFactory {          // BoundTarget -> Executor
    fn make(&self, target: &BoundTarget) -> Result<Executor>;
}
```

The factory is injected; the kernel never constructs concrete adapters itself.
`BoundTarget.transport` picks the arm: `DirectHttp` → `Chat`, `CliWrap` →
`Agent`.

## Behaviour

- **`dispatch(TaskSpec, Vec<BoundTarget>)`** — attempt loop over the chain:
  1. Build an `Executor` for the current target via the factory.
  2. Execute one attempt (`ChatClient::complete` / `Harness::run`), timing it;
     honor `TaskSpec.timeout` and the cancellation token.
  3. Record an `Attempt { target, transport, started_at, duration_ms, outcome }`.
  4. On error, classify to `ErrorKind` and **fail over only when
     `ErrorKind::failover_eligible()` is true AND at least one target remains** —
     set that attempt's `AttemptOutcome::Err.failed_over` accordingly.
  5. Otherwise stop; the terminal status is `Success` / `Error` / `Timeout` /
     `Cancelled`.
- **`RunRecord` assembly:** `run_id` = UUIDv7 (`uuid` v7); `owner` from the
  caller (default `"local"`); `started_at`/`finished_at` bound the whole
  dispatch; `task_digest` per `digest`; `attempts` is the full ordered trail;
  `target`/`transport` are the **last** attempt's; `usage` aggregated via
  `Usage::add`; `status`, `session_id`, `output_chars`, `error`, `labels`
  (copied from the task) populated.
- **`broadcast(TaskSpec, Vec<Vec<BoundTarget>>)`** — run each chain through
  `dispatch` concurrently under a concurrency semaphore (`tokio::sync`), collect
  results **in input order** (index-preserving) despite out-of-order completion.
  Partial failure is normal: some chains end in `Error`, others `Success`, each
  with its own `RunRecord`.
- **Ledger + session after every run.** `Ledger::append` and the session update
  run for **both** success and failure. Session update writes back the harness
  `native_session_id` or the direct-chat transcript and bumps `run_count`.
- **Cancellation.** A `CancellationToken` (re-exported from `vyane-core`)
  propagates into the active attempt; a cancelled run still writes its
  `RunRecord` with `status = Cancelled`.

## Frozen interfaces it composes

- Traits: `ChatClient`, `Harness`, `Ledger` (`append`/`query`), `SessionStore`
  (`load`/`save`/`list`) — all from `vyane-core`, held as trait objects.
- Types: `TaskSpec`, `BoundTarget`, `Target`, `AdapterTransport`, `GenParams`;
  `RunRecord`, `Attempt`, `AttemptOutcome::{Ok, Err}`, `RunStatus`, `Usage`;
  `SessionRecord`; `ErrorKind` / `VyaneError` / `Result`; `CancellationToken`.
- Failover gate is `ErrorKind::failover_eligible()` — the kernel must not
  re-implement the eligibility rule, only call it.

## Acceptance (mechanically checkable)

All via **mock** `ChatClient` / `Harness` / `Ledger` / `SessionStore` and a mock
`ExecutorFactory` — no network, no subprocesses:

- **Failover gating:** a `RateLimited` first target fails over to the second; a
  `Config` first target aborts without a second attempt; a `Cancelled` aborts.
  (Mirror the eligibility table exactly.)
- **Model non-leakage across providers:** given a two-element chain with
  different providers, assert each `Attempt.target.model` is paired only with
  its own `Attempt.target.provider` (no id crosses the boundary).
- **Attempt-trail completeness:** after a fail-over-then-succeed, `attempts` has
  both entries in order, the first with `failed_over = true`, and
  `RunRecord.target` equals the second (successful) target.
- **Broadcast ordering + partial failure:** N chains where some fail and some
  succeed return results in the exact input order, each with its own
  `RunRecord`.
- **Cancellation:** cancelling mid-attempt yields `status = Cancelled` **and** a
  ledger append still occurs.
- **Ledger-write-on-failure:** an all-targets-fail dispatch still calls
  `Ledger::append` exactly once with `status = Error`.

## Non-goals

- No concrete adapters (WP-02 / WP-03 provide them behind traits).
- No config parsing or chain resolution (WP-01 hands the kernel resolved
  `BoundTarget`s).
- No routing/selection policy (`vyane-router`, v0.3).
- No workflow/multi-stage orchestration (v0.2).

## Notes

- **`task_digest` via `sha2`** is the one new workspace dependency this package
  needs, and it is **pre-approved by the orchestrator**. Add `sha2` to
  `[workspace.dependencies]` and depend on it here; do not hand-roll a hash from
  `std` (the field is specified as SHA-256, hex, first 16 chars). This is the
  *only* sanctioned addition to the workspace dep set for this WP — anything
  else needs separate sign-off.
- Aggregate multi-attempt token usage with `Usage::add` rather than summing
  fields by hand.
- Keep the factory injection explicit so tests can supply deterministic mock
  executors; the kernel must remain runtime-free beyond `vyane-core`.
