## SPEC (architect-authored — save as docs/plan/WP-06.md, then implement)

Binary: `vyane`.

Commands (v0.1):
- `vyane check` — load + validate config (report file paths used), list providers/profiles, probe harness availability (claude-code, codex-cli via Harness::available), report which profile env vars are present/missing (names only, never values). Exit 0 even with warnings; exit 2 on config parse errors.
- `vyane dispatch <TASK> --target <profile|provider/model>` with options: --workdir PATH, --sandbox read-only|write|full (default read-only), --session ID, --system TEXT, --timeout SECS (absent = no timeout), --label k=v (repeatable), --json. Resolves the target (profile → full failover chain; provider/model → single-element chain), assembles the kernel, runs, prints the answer text (human) or the full RunRecord + output (--json). Exit 0 on RunStatus::Success, 1 otherwise (error/timeout/cancelled); config-phase failures (unparsable config, unknown profile/target) exit 2, mirroring `check`, so wrappers can tell "fix the config" from "the run failed".
- `vyane broadcast <TASK> --targets a,b,c` (same options minus --session) — concurrent chains, prints per-target results in input order with a compact summary table (target, status, duration, output first line) or --json array.
- `vyane history` — query the ledger: --limit N (default 20), --status, --provider, --json. Human output: one line per run (short run_id, started_at, target, status, duration, cost if present).
- `vyane sessions` — list session records: id, target, run_count, updated_at; --json.

Assembly rules:
- ExecutorFactory impl maps BoundTarget: transport DirectHttp → protocol client by Protocol (OpenaiChat/AnthropicMessages/OpenaiResponses) constructed with the target Endpoint; transport CliWrap → harness by HarnessKind (ClaudeCode/CodexCli), endpoint passed through, env via vyane-config's env-injection builder (EnvPolicy scrubbed + inject set). Unsupported combos → ErrorKind::Unsupported with a message naming the combo.
- Config loading per vyane-config layering (user file then project file), plus --config PATH override that REPLACES the file list.
- Storage paths: ledger at $VYANE_DATA_DIR/ledger.jsonl, sessions at $VYANE_DATA_DIR/sessions/; VYANE_DATA_DIR env overrides; default = dirs::data_dir()/vyane. Create dirs on demand.
- tracing-subscriber env-filter wired to RUST_LOG, default warn.
- Cancellation: ctrl-c (tokio signal) triggers CancellationToken so the RunRecord still lands (kernel guarantees append).

Acceptance:
- assert_cmd integration tests: `vyane check` with a temp config exits 0 and lists the profile; `vyane dispatch` end-to-end against a wiremock OpenAI-chat endpoint using a temp config + temp VYANE_DATA_DIR — asserts the answer prints, exit code 0, AND the ledger file contains exactly one RunRecord with status success (parse the JSONL); a failing dispatch (wiremock 500s with failover to a second wiremock target that succeeds) proves chain failover end-to-end through the CLI; `vyane history` then shows the runs; --json outputs parse as JSON.
- No real network, no real CLIs in tests.
- Help text (--help) is polished — this is the public front door.

Non-goals: no workflow command (v0.2), no daemon, no router integration beyond config-declared chains, no colors/TUI.
