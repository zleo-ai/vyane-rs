# WP-41 — Live authority at native side-effect boundaries

## Goal

Add the first native model and tool entry points that cannot silently bypass a
live execution-authority check. The work establishes a fail-closed contract and
guarded consumers without registering a `NativeHarness` or claiming a complete
native execution path.

Integrated into the current clean public baseline.

## Authority vocabulary

- `NativeExecutionAuthority` is an object-safe, process-local async contract.
  A successful revalidation grants only the exact `NativeSideEffect` passed to
  that call and cannot be cached for a later effect.
- `NativeSideEffect` identifies model sends, tool operations, checkpoint
  prepare/publish points, and a revision-fenced session commit using ordering
  coordinates only. It carries no prompt, request body, tool arguments, path,
  credential, token, or native session id and has no Serde implementation.
- `AuthorizedToolChatClient` is deliberately independent from `ChatClient`.
  A native-loop caller holding the narrower trait object has no unguarded
  `complete_turn` fallback on that object.

## Atomic AgentRun validation

- `NativeExecutionScope` freezes target key, prompt digest, policy digest,
  optional logical session, and either fresh execution or one opaque resume-
  binding proof. Its debug representation hides digests/proof and it is not
  serializable.
- `AgentStore::validate_native_execution_permit` defaults to fail closed.
- `SqliteAgentStore` validates the scope and `ActiveExecutionPermit` in one
  write-locked snapshot. The predicate includes owner, run/worker identity,
  worker generation, lease owner and secret capability, running/open
  lifecycle, lease/deadline, policy, target, prompt, logical session and exact
  fresh-versus-resumed binding.
- Heartbeat and activity writes may advance revisions without revoking a
  still-matching permit. Cancellation, terminal state, generation/token drift,
  lease/deadline expiry, policy drift, or any native-scope drift fails closed.

This store operation does not by itself implement
`NativeExecutionAuthority`; a bridge has to own the permit and scope and call
the store for each effect.

## Guarded OpenAI Chat sends

- The authorized non-streaming OpenAI Chat typed-turn path validates the typed
  request before consuming authority, then revalidates immediately before each
  explicit physical `send` with a 1-based wire-attempt coordinate.
- Authority errors are returned unchanged and never enter the HTTP retry
  policy. Transport/status retries return to the explicit loop and require a
  new authority check.
- After typed-request validation, cancellation is observed before request
  construction, while authority is pending, after it returns, during `send`,
  while reading a successful body, and during retry backoff.
- The shared reqwest client follows no redirects and performs no implicit
  retries. The explicit Vyane loop is therefore the sole owner of attempt
  numbering and guarded sends.

The ordinary `ChatClient` methods remain available for compatibility and are
not native-execution-authority consumers. No production native loop calls the
authorized trait yet.

## Guarded tool dispatch

- `ToolRegistry::execute_authorized` evaluates validation and permission first.
  Only an allowed call consumes `ToolOperation` authority, immediately before
  its executor is polled.
- Invalid, unknown, denied, approval-required, already-cancelled and already-
  expired outcomes remain pure decisions and consume no authority.
- Cancellation/deadline races remain live while authority is pending and are
  checked again before executor polling. Authority failure is an outer
  execution error, never bounded model-facing tool output.
- The older `ToolRegistry::execute` remains a compatibility API and is not an
  authority consumer. A production native loop must use the guarded entry.

The registry gate is one dispatch linearization point, not blanket authority
for arbitrary tool code. A trusted built-in that performs multiple opens,
publishes, network calls, or process spawns must revalidate every additional
external operation before it becomes visible.

## Acceptance

- Native-scope contract tests reject target, prompt, policy, logical-session,
  resume-proof, fresh/resumed and permit-state drift while preserving expected
  heartbeat/activity behavior and redacted debug output.
- Fake HTTP servers prove `N` explicit sends require `N` authority checks,
  revocation before retry prevents the next request, authority failures are not
  retried, redirects are not followed, and cancellation stops authority wait,
  in-flight send, response-body read and backoff.
- Tool-registry tests prove exact effect coordinates, no executor call after
  revocation, zero authority consumption for pure decisions, and cancellation
  winning before executor polling.
- Workspace format, Clippy-with-warnings-denied, tests, and staged secret scans
  are required by the repository merge gate.

## Integration state and non-goals

At its delivered baseline, WP-41 is a dark authority seam, not a native
execution milestone. It does not provide:

- a concrete `ActiveExecutionPermit`/store implementation of
  `NativeExecutionAuthority`;
- composition with `SessionExecutionLease` and exact `NativeSessionDomain`
  validation;
- a bounded model-to-tool-to-model driver or registered `NativeHarness`;
- trusted read/write/bash/search tools, per-operation filesystem authority, or
  host-enforced sandboxing;
- approval persistence/one-shot consumption;
- checkpoint preparation/publication or revision-fenced session commit;
- production native resume, resident AgentRun supervision, or automatic
  recovery/replay.

Those boundaries remain fail closed. Native resume can be enabled only after a
session-aware production authority composes AgentRun and session/domain truth
and the complete side-effect chain has executable drift, cancellation, retry
and restart acceptance coverage.

## Follow-on status

The integrated baseline adds `AgentRunModelToolAuthority`, the first concrete bridge
from an owned `ActiveExecutionPermit`, `NativeExecutionScope`, and `AgentStore`
to `NativeExecutionAuthority`. It accepts only fresh sessionless scopes and
only authorizes one-based model sends or tool operations, with each synchronous
store validation moved to Tokio's blocking pool. It rejects session-bearing
scopes, checkpoint prepare/publish and session commit. No production factory,
runtime, or native loop constructs it, so all other non-goals above remain
open. The bounded bridge and the separate AgentEvent projector are recorded in
[WP-42](WP-42.md).
