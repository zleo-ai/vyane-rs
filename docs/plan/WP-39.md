# WP-39 — Durable task domain and private worker handoff

## Goal

Unify REST asynchronous tasks and CLI detached tasks on one durable,
metadata-only task domain while ensuring a detached request is never frozen in
the task store, argv, or environment.

SQLite is the sole lifecycle truth for every new task. Process-local state is
limited to live cancellation tokens and opaque runtime task handles;
filesystem task directories contain only private operational/result artifacts.
Older `job.json` / `status.json` directories stay readable through an isolated
compatibility path and are never written by a new submission.

## Shared task domain

- The standalone `vyane-task` crate owns `TaskRecord`, `NewTask`, lifecycle
  state, controller identity, failure codes, leases, queries, events, and the
  synchronous `TaskStore` contract.
- Canonical states are `queued`, `running`, `cancelling`, `succeeded`, `failed`,
  `timed_out`, `cancelled`, and `interrupted`.
- A controller is either an in-process instance id or a detached process-group
  identity. Cancellation tokens are not serializable and live only in a map
  keyed by `(task_id, executor_epoch)`.
- `NewTask` has no prompt, system instruction, session content, arbitrary
  labels, output, raw error, endpoint, environment, or credential field. This
  is a structural boundary rather than content classification: front-ends must
  derive identifiers and digests and must not relabel private content as
  metadata. Free-form Unicode target/profile names remain compatible with
  `vyane-config`; task digests use lowercase hexadecimal.
- Attaching or claiming a controller advances `executor_epoch`. Every mutation
  compares both revision and executor epoch so a stale worker or lease owner
  cannot settle a newer execution.
- Exact terminal settlement retry is idempotent with the current revision or
  the immediately preceding revision from a lost commit response, provided the
  executor epoch and terminal result match.
- Requested wall-clock timestamps are normalized to milliseconds and clamped
  to the snapshot's last update. Clock rollback therefore cannot block cancel
  or settlement, and lease expiry is validated against the clamped effective
  time.

## SQLite store

- `$VYANE_DATA_DIR/tasks.sqlite3` stores task snapshots and append-only metadata
  events. Snapshot mutation and event append occur in one IMMEDIATE transaction.
- The bundled `rusqlite` backend enables WAL, foreign keys, a five-second busy
  timeout, and `synchronous=FULL`.
- On Unix, a new database is prepared privately at mode `0600` and published
  create-only after its creator descriptor closes; SQLite-created WAL/shared-
  memory sidecars must also have mode `0600`. Existing SQLite-managed files are
  checked without reopening them; a symlink, non-regular file, or non-`0600`
  mode fails closed and must be repaired while the store is offline.
- Listing is stable and descending by `(created_at, id)`, with owner/state
  filters, a bounded limit, and cursor pagination.
- Expired lease claim advances executor epoch. Lease renewal requires the same
  owner, an unexpired current lease, and a strictly later expiry.
- `PRAGMA user_version` controls migrations. A newer schema is rejected without
  downgrade both before opening and again after every writer obtains
  `BEGIN IMMEDIATE`, closing the rolling-upgrade race. A matching version must
  have the exact migration-defined table, constraint, foreign-key, and index
  structure or open fails.
- SQLite type mismatches and invalid stored task/event domain invariants surface
  as `CorruptData`. Only a confirmed duplicate task id maps to `AlreadyExists`.

## REST integration

- `POST /v1/tasks` creates durable `queued` metadata, attaches an in-process
  controller, registers its live token, and then spawns supervised dispatch.
- `GET /v1/tasks` and `GET /v1/tasks/{id}` read SQLite. Their bounded wire shape
  contains digest, target, state, timestamps, revision, ledger run id, and
  failure code — never prompt, output, or raw errors.
- Successful asynchronous output is retained separately at
  `$VYANE_DATA_DIR/tasks/<id>/output.txt` with mode `0600` and served by
  `GET /v1/tasks/{id}/output`. The SQLite row contains no output body, and a
  non-successful or missing artifact produces a bounded API error.
- Cancellation first performs the durable state transition and then signals the
  matching live token. Completion remains authoritative after a cancellation
  race.
- A nested dispatch panic is converted to bounded `FailureCode::Internal`.
  Completed dispatches retain their exact-epoch token and tracked Tokio handle
  while settlement retries with bounded backoff; ownership is removed only
  after durable settlement or shutdown abort.
- `vyane serve` binds before recovery, obtains an exclusive per-data-directory
  supervisor lock, then marks abandoned REST `queued`, `running`, and
  `cancelling` rows `interrupted`. It never claims detached CLI tasks.
- Graceful server shutdown closes task admission, durably requests cancellation,
  signals exact-epoch tokens, and performs a bounded drain while the supervisor
  lock is still held. A timed-out worker is interrupted only after an exact
  instance/epoch ownership re-check. Any metadata-store error still enters a
  finally path that waits initializers, cancels tokens, aborts and awaits every
  tracked dispatch, and only then returns the first error and releases the lock.
- Router construction for embedding does not mutate old rows. Listing and
  recovery traverse all SQLite cursor pages rather than stopping at 1,000 rows.

## Detached CLI integration

- The parent validates routing, creates a durable `queued` row, serializes a
  versioned `WorkerEnvelope` in memory, and spawns
  `vyane __worker <task-id>` with piped stdin.
- The task id is the only request-specific argv value. Prompt, system text,
  labels, session, config override, workdir, and route snapshot are absent from
  argv and environment. The envelope is bounded to 32 MiB and EOF marks a
  complete handoff.
- A new task directory may contain mode-`0600` `task.log`, `output.txt`, an
  ephemeral `harness-controller.json`, and its mode-`0600` advisory lock. The
  controller sidecar contains only PID/PGID/start/birth identity for a controlled
  shell sentinel. Started/Stopped operations are serialized across processes
  with a 500 ms lock budget, use unique create-new temp files, and remove only
  the exact birth after the entire group is observed empty. A new Started is
  rejected while a different pending controller remains, so failover cannot
  erase the only recovery record. A parent-held stdin
  start gate prevents the real CLI from executing until Started commits; parent
  death while publication is blocked closes the gate without launching the CLI.
  The sentinel remains the group leader while the real target runs as its child,
  returns the target exit code over a private pipe, and kills its own whole group
  before exiting. None of these files or pipes contains a request or result.
  The directory contains neither `job.json` nor `status.json`; SQLite remains
  the only lifecycle truth.
- If spawn or stdin handoff fails, the parent kills and reaps the worker process
  group and settles the durable row as `SpawnFailed`. An attach-before-EPIPE
  race is settled only when the reaped child's PID, PGID, and birth fingerprint
  match the recorded controller; a newer executor is never borrowed. The
  transport layer does not create a fallback status file.
- The worker arms SIGTERM cancellation before attaching its process controller.
  On Linux the controller records the process birth fingerprint in addition to
  pid, pgid, and start time. PID liveness, exact PGID, and the kernel fingerprint
  are authoritative, so clock changes and a missing `ps` binary cannot reject
  the same worker; unavailable or mismatched fingerprint evidence fails closed.
- Status/list reads never infer worker loss from queued age or a temporarily
  unavailable identity probe. They persist `interrupted` only for confirmed
  death or a proven identity mismatch, and preserve an active control entry
  when an outer-dead task still has a possibly live nested sidecar. Explicit
  cancellation consumes an exact nested sidecar before terminalizing, including
  when metadata was already terminal. A terminal row with a still-exact live
  outer controller is force-cleaned too, so a stopped worker cannot retain an
  unreaped nested zombie. It revalidates the outer controller before TERM and
  again before KILL. Once Started is gated and published, only the still-exact
  sentinel authorizes group signalling. A dead sentinel plus a live numeric
  PGID is ambiguous and fails closed instead of risking a reuse mis-kill.
- CLI harnesses keep their own process groups. A fallible runtime lifecycle
  reporter publishes the exact nested controller before execution may proceed;
  report failure kills and reaps the fresh harness. A normally exiting leader
  cannot publish `Stopped` until closed-stdio background descendants that remain
  in its process group are also cleaned. Forced cancellation verifies and
  signals both outer and nested groups, including when a stopped worker cannot
  forward its cancellation token; nested cleanup and outer settlement have
  distinct grace budgets.
- Success writes the optional result artifact through a unique-temp atomic
  replacement and settles SQLite with its ledger run id. An I/O failure cannot
  expose partial output. Full diagnostics remain in the private log; only
  bounded failure codes enter task metadata.
- Empty worker stdin selects read-only compatibility for a pre-WP-39
  `job.json`. Legacy status/output behavior remains available without becoming
  a second writer for new tasks.

## Acceptance

- Store reopen preserves list/get snapshots and event history.
- Create-to-attach advances revision and executor epoch; illegal transitions,
  stale epoch writes, and conflicting terminal results are rejected atomically.
- Cancellation versus settlement produces one CAS winner and can still settle
  an authoritative result from `cancelling`.
- Injected event failure rolls back the snapshot. A real child process that
  aborts after uncommitted snapshot/event writes leaves both unchanged after
  reopen, and the next legal transition succeeds.
- Two OS child processes using the same attach CAS produce exactly one winner.
- Clock rollback is clamped consistently for updated/event/finished timestamps,
  and cannot create an already-expired attach, claim, or renewed lease.
- Cursor pagination, expired lease claim, monotonic renewal, nanosecond input
  normalization, write-lock-time forward-schema rejection, exact schema
  validation, and stored-type/domain corruption classification have contracts.
- Database/WAL bytes contain none of the payload canaries. On Unix the
  database/WAL/SHM permissions are `0600`.
- Normal, auto-routed, frontier-filtered, cancelled, failed-handoff, and orphaned
  detached runs use SQLite and create no request/status JSON files.
- A slow queued attach and an unavailable process-identity probe remain
  non-destructive under list/status reads; queued rows are explicitly
  cancellable by CAS.
- A real stopped detached worker with a TERM-ignoring nested harness is forced
  down without leaving either process group alive.
- A worker killed with SIGKILL leaves its independently grouped harness alive
  until explicit cancel; direct cancel, status-then-cancel, and already-terminal
  metadata paths all verify and remove that orphaned group.
- A blocked Started publication never releases the CLI start gate. A dead nested
  sentinel never authorizes a live numeric PGID, while an outer crash during a
  real run leaves the exact sentinel available for recovery. Terminal metadata
  over a stopped outer worker and terminalization racing an in-flight cancel
  both clean the exact groups. A wedged controller lock returns within its
  bounded budget.
- A responsive nested harness receives graceful cancellation through the
  worker-owned token and is recorded as `cancelled`, not raced into a generic
  harness failure by a second direct TERM.
- A harness leader that exits after launching a closed-stdio background process
  cleans descendants remaining in its process group before reporting `Stopped`,
  for capture and stream.
- REST completed rows survive router/server reconstruction; startup recovery,
  panic settlement, retrying transient settlement failures, duplicate-epoch
  rejection, cancellation-token cleanup, exclusive serve ownership, metadata
  error cleanup, and multi-page traversal are covered. Successful output
  survives restart through the private artifact endpoint while the SQLite/WAL
  privacy scan remains free of output canaries.
- The 13-crate publish order includes `vyane-task`, and its packaged artifact
  contains the SQL migration.

## Non-goals

- Persisting or replaying task request payloads after a process crash.
- Pause/resume semantics.
- A resident daemon or lease-renewal loop.
- Protection from a deliberately hostile same-UID child. The sentinel closes
  ordinary crash/cancellation windows and fails closed if destroyed; hostile
  workloads require an OS/container/VM isolation boundary.
- Migrating the run ledger, sessions, workflow journals, logs, or result
  artifacts into SQLite.
- Removing the read-only legacy `job.json` / `status.json` compatibility layer.
