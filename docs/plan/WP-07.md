## SPEC (architect-authored — save as docs/plan/WP-07.md, then implement)

A declarative multi-step workflow: a TOML file describing a DAG of dispatches. Forward-compatible with a future scripted engine; deliberately boring now.

File format (workflow.toml):
- [workflow]: name (required), description?, max_concurrency (default 4).
- [[step]]: id (unique, required); needs = ["step-ids"] (default []); exactly ONE of target = "profile|provider/model" OR fan_out = ["t1","t2",...]; prompt (template string) XOR prompt_file (path relative to the workflow file); system?; workdir?; sandbox? (read-only default); timeout_secs?; on_error = "abort" (default) | "continue".

Templating (own tiny implementation, no new deps): {{steps.<id>.output}} — single-target step's answer text; {{steps.<id>.outputs}} — fan_out step: all answers concatenated with "## <target>\n" headers; {{vars.<key>}} — from CLI --var k=v; {{workflow.name}}. Unknown placeholder → validation error. Escape rule: literal {{ via {{{{ (document it).

Validation (ALL before any dispatch, collecting every problem into one error report): duplicate/unknown step ids, dependency cycles (topo sort), unknown template refs (a step may only reference steps in its transitive needs), both-or-neither target/fan_out, prompt XOR prompt_file, unresolvable targets (resolution happens through an injected resolver — see below).

Execution semantics:
- Topological scheduling: a step becomes ready when all needs succeeded; ready steps run concurrently bounded by max_concurrency (tokio semaphore).
- Single-target step = one kernel dispatch with the config-resolved failover chain. fan_out step = kernel broadcast over the resolved chains; step output = ordered outcomes; step SUCCEEDS if at least one target succeeded, and {{steps.X.outputs}} includes only successful answers (failed targets noted in the journal).
- on_error = "abort": stop scheduling new steps, let in-flight finish, workflow status = failed. "continue": mark step failed, transitively mark dependents Skipped, independent branches proceed; workflow status = completed_with_failures.
- Cancellation (ctrl-c via CancellationToken): cancel in-flight dispatches, journal everything, workflow status = cancelled.

Journal & resume:
- Journal file per workflow run at $VYANE_DATA_DIR/workflows/<wf_run_id>.json (uuidv7), atomic tmp+rename rewrites after every step transition: {wf_run_id, workflow_name, file_sha256, vars, started_at, updated_at, status, steps: {id: {status: pending|running|success|failed|skipped|cancelled, run_ids: [...], output: Option<String>, outputs: Option<Vec<{target, ok, output?}>>, error?}}}.
- The journal stores step outputs in full — it is local working state, unlike the ledger (which stays digest-only). Document this distinction.
- `vyane workflow resume <wf_run_id>`: reload journal; refuse if the workflow file's sha256 changed (clear error); steps with status=success are skipped reusing recorded outputs; failed/skipped/pending/cancelled re-run per normal scheduling. vars come from the journal (CLI --var on resume = error).

Crate architecture:
- vyane-workflow depends on vyane-core + vyane-kernel + serde/serde_json/toml/thiserror/tokio/tokio-util/futures/tracing/uuid/chrono/sha2 (all already in workspace deps). It must NOT depend on vyane-config: target resolution is injected via `trait TargetResolver: Send + Sync { fn resolve(&self, target: &str) -> vyane_core::Result<Vec<BoundTarget>>; }` defined in vyane-workflow, implemented in the CLI over ResolvedConfig.
- Engine entry: `WorkflowEngine::new(dispatcher: Arc<Dispatcher>, resolver: Arc<dyn TargetResolver>, journal_dir: PathBuf)`; `run(&self, wf: &Workflow, vars, cancel) -> WorkflowOutcome`; `resume(&self, wf_run_id, wf: &Workflow, cancel) -> WorkflowOutcome`.

CLI (crates/vyane-cli):
- `vyane workflow run <FILE> [--var k=v]... [--json]` — human output: live per-step status lines (started/succeeded/failed with duration) then a summary table; exit 0 only when status == completed (all steps success), 1 otherwise, 2 on validation/config errors (consistent with existing convention).
- `vyane workflow resume <WF_RUN_ID> --file <FILE> [--json]`
- `vyane workflow list [--json]` — journals most-recent-first: id, name, status, started_at, step counts.

Acceptance (mock-based in vyane-workflow via mock Dispatcher path — note Dispatcher is concrete; test through a mock ExecutorFactory injected into a real Dispatcher, the same pattern vyane-kernel tests use):
- validation: cycle detection, unknown ref, both target+fan_out, unknown template var — each reports ALL errors at once;
- scheduling: diamond DAG (a→b,c→d) runs b and c concurrently (prove overlap with delayed mocks + high-water counter), d sees both outputs;
- templating: substitution correctness incl. fan_out headers + vars + escape;
- on_error both modes (abort stops scheduling; continue skips dependents, independent branch completes);
- fan_out partial failure: step succeeds, outputs contain only successes;
- cancellation mid-flight: journal has cancelled statuses;
- resume: completed steps not re-dispatched (mock counts calls), outputs reused, changed-file-hash refused;
- CLI: one assert_cmd end-to-end with wiremock (two-step workflow: draft → review referencing {{steps.draft.output}}), journal file asserted, exit codes; `workflow list` shows it.

Non-goals: no scripted/JS engine, no cron/scheduling, no per-step retry policies beyond the kernel's failover, no dynamic DAG mutation.
