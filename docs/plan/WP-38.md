# WP-38 — Executable auto-routing

## Goal

Close the gap between routing preview and execution without teaching the
kernel about profiles. `vyane-service` owns the boundary:

```text
rendered TaskSpec → RouteDecision → profile → BoundTarget chain → Dispatcher
```

## Contract

- `RouteDecision.provider` is always a real provider id. The selected config
  profile is carried separately as `RouteResult.profile`; an opaque selection
  key preserves exact identity when profiles share provider/model.
- `--target auto` works through CLI, REST/MCP dispatch, REST SSE, detached
  dispatch, and a single-target workflow step.
- The normalized route effort overrides every leg in the selected failover
  chain and reaches both direct HTTP requests and Claude Code/Codex CLI argv.
- `--no-frontier` or `routing.allow_frontier=false` prevents a frontier target
  or failover leg from being executed even when task text/tags request it; no
  safe candidate is an error, never a provider fallback.
  Profile-named failover legs are classified by that exact profile's tier and
  frontier legs are removed. A raw `provider/model` failover is retained only
  when configured profile metadata proves the tuple exclusively economy or
  mainline; unknown or ambiguous tuples fail closed before dispatch.
- Auto decisions are recorded as `routing.*` labels on every resulting
  `RunRecord`; prompt text is still not persisted in the ledger.
- Detached execution freezes the selected profile, canonical routing labels,
  and a secret-free target snapshot before re-exec. The worker reapplies effort
  and fails closed if config drift changes the resolved chain.
- Workflow `target = "auto"` is deferred until the prompt template has rendered.
  `[step.route]` accepts `stage`, `tier`, `tags`, `candidates`, and
  `allow_frontier`. Deferred routing in `fan_out` is intentionally rejected
  until per-target task metadata exists.
- Routing decision labels are reserved against caller forgery. A literal
  profile named `auto` is explicitly selectable as `profile:auto`; the
  `target:` escape preserves provider/model selectors that share the prefix.
- Workflow identity includes referenced `prompt_file` content, so resume cannot
  silently accept a changed external prompt.

## Acceptance

- Economy/frontier routes execute different configured models and effort.
- Frontier denial falls back to a non-frontier profile and truthful effort.
- Provider and profile identities never share the same field.
- Claude Code receives `--effort`; Codex receives a two-token
  `-c model_reasoning_effort="..."` override before `exec`.
- Online, streaming, detached, and workflow paths retain route labels.
- Existing explicit selectors and no-budget/no-route workflows remain
  behaviorally unchanged.
