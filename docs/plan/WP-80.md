# WP-80 · Owner-frozen goal continuity read service

## Outcome

Expose the WP-79 continuity next-action projection through one explicitly
enabled, bearer-authenticated loopback REST read path. The service freezes one
authority-supplied owner when it is constructed and returns an allowlisted,
redacted view of one atomic goal-and-approval snapshot. The bundled REST
assembler supplies the fixed `local` owner.

## Contract

- `GoalReadService` is an opt-in component. Ordinary service construction does
  not open the goal database; a protocol assembler must explicitly construct
  the reader before it can register the route.
- Construction freezes the goal store and an owner previously derived from an
  authenticated principal for the lifetime of the component. The bundled REST
  assembler deliberately supplies `local`; the request cannot select or
  override that owner through a path, query, header or body field.
- The REST assembler explicitly constructs the component and registers only
  `GET /v1/goals/{id}/continuity-next` on the existing loopback server. It
  requires that server's
  per-start bearer capability and inherits its loopback `Host`/`Origin` and
  cross-site request checks.
- One read transaction observes the exact goal snapshot and all approval rows
  used by the projection. A response cannot combine a goal revision with
  approval evidence from another revision.
- Foreign and absent goal ids remain indistinguishable. Malformed ids and
  inconsistent or missing approval evidence fail closed through bounded,
  stable errors.
- The success DTO is an explicit allowlist containing only the identifiers,
  revision/quota boundary and closed next-action fields needed by an operator.
  It excludes goal title and acceptance detail, progress bodies, prompts,
  workdir and sandbox paths, target configuration, raw approval/run records,
  artifacts, output and raw errors.
- Repeating the same request against unchanged durable state returns the same
  representation and performs no database or filesystem writes.

## Wire contract

The endpoint accepts no query parameters or request body. A successful response
uses `snake_case` fields and the following envelope:

```json
{
  "next_action": {
    "view_schema": 1,
    "goal_id": "goal-123",
    "goal_revision": 7,
    "quota_event_id": "quota-123",
    "action": "queue_approval",
    "command": "continuity_queue",
    "step_id": "takeover",
    "step_kind": "start_takeover",
    "required_inputs": ["workdir", "sandbox", "timeout_seconds"],
    "reason_code": "approval_required"
  }
}
```

Optional fields are omitted when they do not apply. `action`, `command`,
`accepted_signals` and `reason_code` are closed enums represented as
`snake_case` strings. Consumers must fail closed on an unknown `view_schema` or
enum value instead of guessing a mutation or execution action.

| Status | Meaning |
| --- | --- |
| `200` | One coherent allowlisted projection. |
| `400` | Malformed goal id, any query parameter or a non-empty request body. |
| `401` | Missing or incorrect bearer capability. |
| `403` | Non-loopback Host/Origin or cross-site request. |
| `404` | The goal is absent from the frozen owner scope, including a foreign-only id. |
| `405` | Any method other than `GET`; the response carries `Allow: GET`. |
| `409` | The goal exists but has no valid continuity projection. |
| `500` | The store or blocking read worker is unavailable; internal details are redacted. |

Every response for this resource, including authentication, authority and
method failures, carries `Cache-Control: no-store`.

## Verification gate

- Service tests prove the owner is frozen at construction, cannot be supplied
  by a request and is passed to every store read.
- Store/service tests interleave goal and approval mutations around the read
  transaction and prove each response comes from one coherent snapshot.
- REST acceptance covers bearer rejection, loopback request enforcement, exact
  success JSON, absent/foreign equivalence, malformed input and redaction of
  every excluded field. A service-construction test proves the goal database is
  not opened before explicit reader construction.
- A write detector proves successful and failed requests do not change goal,
  approval or event state.
- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`

## Explicit exclusions

- no goal list or general goal get endpoint;
- no MCP surface;
- no queue, decision, execution, signal, retry, repair or runner operation;
- no automatic goal completion or acceptance mutation;
- no runtime-native session resume;
- no distinct-principal authentication, hostile multi-user isolation or
  hostile same-UID isolation. The bearer protects one fixed local scope; it is
  not a user identity.

## Dependency boundary

WP-80 is the read-only service prerequisite for a later WP-82 runner. WP-81 may
add typed observation ingestion through a separate mutation authority, but it
must not add writes to `GoalReadService` or infer external facts through this
GET endpoint. WP-82 may consume the stable projection only under separately
authenticated and explicitly configured queue/decision/execution authorities;
it must not turn this bearer or this read service into execution authority.

CON-07 and INT-02 remain `partial` after this package.
