# WP-72 · Opt-in resident goal pursuit

Status: in progress

## Goal

Compose the existing lease-fenced `GoalPursuer` into the resident daemon as an
explicitly enabled, bounded, single-owner supervisor. A daemon restart may
continue unfinished automatic work from the durable WP-71 checkpoint without
turning a semantic pause into an implicit resume.

## Contract

- Automatic pursuit is disabled unless `daemon run` or `daemon start` receives
  `--goal-auto-pursue` together with one target and one canonical workdir.
- The resident supervisor handles at most one `local` goal at a time. It first
  adopts eligible `in_progress` work, then atomically claims the next queued
  goal by the store's priority order.
- A stable daemon-goal worker identity is safe because the existing daemon
  supervisor lock excludes a concurrent resident daemon. It lets a replacement
  daemon renew the still-live lease left by its crashed predecessor; expired
  foreign leases still require the normal fenced reclaim path.
- Every runtime segment remains a fresh production dispatch through the
  configured target and explicit sandbox. The verifier, segment, failure and
  invocation budgets remain the WP-70/WP-71 bounds.
- Shutdown cancels the resident runtime and leaves a running checkpoint for
  replay. It does not rewrite interruption as a semantic goal pause.
- Pursuit-level infrastructure errors use bounded exponential retry. After five
  consecutive errors in one daemon generation, that goal is quarantined until
  daemon restart so other queued goals can continue without rewriting goal
  lifecycle truth.
- `Paused`, terminal and actively foreign-owned goals are never auto-resumed or
  stolen. A manual operator action remains required to make a paused goal
  eligible again.

## Explicit limits

- No quota/continuity handoff, approval authority, authenticated goal API,
  remote owner, board/network verifier or runtime-native session resume.
- Configuration is daemon-wide, not persisted per goal. Existing checkpoint
  counters and evidence are adopted, while the current explicit daemon target,
  workdir and sandbox remain the execution authority.
- This slice does not promise parallel goal execution or cross-daemon leasing.

## Verification gate

- Store/controller tests for queued selection, unleased and expired recovery,
  live stable-worker restart adoption, foreign-lease exclusion and paused-goal
  exclusion.
- Cancellation test proving shutdown leaves restartable state and does not
  start a second segment.
- CLI acceptance proving detached `daemon start` forwards the opt-in contract
  and a queued goal reaches its verified terminal state through a fake harness.
- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`

CON-07 remains `partial`: resident automatic pursuit closes the local daemon
assembly gap, while approval/quota handoff and authenticated service exposure
remain separate work.
