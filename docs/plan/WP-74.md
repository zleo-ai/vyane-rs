# WP-74 · Explicit controlled takeover

## Goal

Close one narrow continuity layer: turn the current ready takeover step into a
durable approval, record one explicit operator decision, and consume an
approved boundary for exactly one dispatch.

## Contract

- Queueing is owner-scoped and only accepts the current
  `takeover` / `start_takeover` step of an `in_progress` goal.
- The approval binds the quota event, full typed target axes, goal revision,
  continuity snapshot, canonical work directory, sandbox and timeout.
- Queueing the same snapshot is idempotent and never resolves or starts a
  runtime.
- Approve and reject are explicit durable decisions. A decided approval cannot
  be overwritten.
- Execution accepts only an approval id. It re-reads the current goal and fails
  closed if the goal, step, quota event, target or execution boundary changed.
- Target resolution must produce one target whose provider, protocol, harness
  and model exactly match the approval. Profile failover chains are rejected.
- Approval consumption and the visible `in_flight` step transition share one
  SQLite transaction before dispatch. A crash cannot make the approval
  executable a second time.
- Completion records `done` or `blocked` plus bounded run-status evidence in
  the approval row and visible goal progress.

## CLI

```text
vyane goal continuity-queue GOAL_ID --workdir PATH [--sandbox write] [--timeout-seconds 300]
vyane goal continuity-decide APPROVAL_ID --decision approve|reject --decided-by OPERATOR
vyane goal continuity-execute APPROVAL_ID
```

The execute command has no target, workdir, sandbox or timeout override.

## Deliberate exclusions

- no automatic approval or execution;
- no quota-ledger reader or periodic continuity runner;
- no review, repair, handback or primary resume execution;
- no HTTP, MCP or authenticated goal service;
- no retry of a consumed approval and no autonomous failover.

## Parity boundary

CON-07 remains `partial`. This work package closes the explicit approval and
one-shot takeover boundary. Review/handback, external quota ingestion and an
authenticated goal service remain separate future layers.
