# WP-76 · Bounded quota reader and explicit reset signal

## Outcome

Add a platform-neutral upstream quota snapshot substrate and one explicit,
typed `quota_reset` continuity signal. The reader normalizes a finite connector
set without embedding provider endpoints or credentials. The goal signal only
releases the visible quota dependency; it never consumes an approval, resumes a
runtime or dispatches work.

## Quota reader contract

- `QuotaConnector` exposes a stable connector/provider identity and returns one
  normalized `QuotaCard` containing a closed status, bounded windows and an
  optional absolute balance.
- `QuotaSnapshotRunner` validates connector identity and policy before I/O,
  rejects duplicate IDs, bounds concurrency and connector count, applies a
  whole-operation timeout, isolates connector failures and sorts results by
  connector ID.
- Connector errors cross the boundary only as closed redacted codes. Returned
  cards must match the exact connector/provider identity and pass schema
  validation.
- `QuotaHttpReader` accepts only HTTP(S), rejects redirects, caps the complete
  response body and bounds the whole request/body operation by one timeout.
- The crate contains no provider endpoint, credential loader, periodic watcher,
  persistence or automatic routing action. Acceptance tests use fakes and a
  loopback mock server only.

## Goal signal contract

- `vyane goal continuity-signal GOAL_ID quota-reset` requires the exact current
  quota event ID plus primary provider, harness and model, and a bounded source
  label.
- The goal must be owner-visible, `in_progress`, and have a current continuity
  state. Foreign owners, terminal goals, missing state and any boundary mismatch
  fail closed.
- The first exact signal is persisted atomically with one immutable progress
  event. Repeating the same typed signal is idempotent; different evidence for
  an already-recorded signal kind is rejected.
- If review completed first, quota reset makes `resume_primary` ready. If quota
  reset arrived first, the step waits for review. Arrival order does not change
  the final state.
- Empty signal history is omitted from serialization so existing approval
  snapshot digests remain stable.

## Explicit exclusions

- no concrete provider/platform connector or credential source
- no durable quota history, polling daemon, notification or automatic dispatch
- no primary-resume approval or execution
- no authenticated HTTP/MCP goal service

CON-07 and OBS-06 remain `partial` after this package.
