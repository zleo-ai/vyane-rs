# WP-09 — Responses streaming + CLI `--stream`

Maps to roadmap **M9**. Crates: `vyane-protocol`, `vyane-cli`.

## Goal

Fill the one deliberate streaming gap left by WP-02: implement
`ChatClient::stream` for the OpenAI Responses client (`/v1/responses`),
normalized to the same `StreamEvent` contract the Chat and Anthropic clients
already produce. Then expose it at the CLI: `vyane dispatch --stream` prints
deltas to stdout as they arrive for single-target direct-HTTP dispatches, and
still finalizes exactly one `RunRecord` through the ledger.

## Scope (modules)

- `vyane-protocol::sse` — add `StreamProtocol::OpenaiResponses` alongside the
  existing `OpenAiChat` / `Anthropic` variants; a new frame parser for the
  Responses SSE event shape. Reuses the existing byte-buffer decoder
  (`SseDecoder`/`SseStream`) — no new decoder, no new dependency.
- `vyane-protocol::openai_responses` — override `ChatClient::stream` (was the
  default `Unsupported` trait impl); sets `"stream": true` on the wire request,
  same shape as the Chat/Anthropic clients' `stream()` methods.
- `vyane-cli::cli` — `--stream` flag on `dispatch` (dispatch only; not
  broadcast).
- `vyane-cli::command` — the streaming path: resolve the chain exactly as
  today, but when `--stream` is set **and** the resolved chain is a single
  `DirectHttp` target, build the protocol client directly (reusing
  `AssemblerFactory`'s existing `Protocol → concrete client` mapping) and drive
  it as a stream instead of calling `Dispatcher::dispatch`.

## Wire shape: OpenAI Responses SSE

Responses streaming uses **named SSE events** (`event: <type>` line + a `data:`
JSON payload), unlike Chat Completions' bare `data: {...}` frames. The decoder
already collects all `data:` lines per frame (`collect_data_lines`); this work
package additionally reads the JSON payload's own `"type"` field (the API
duplicates the event name into the payload), so no `event:` line parsing is
needed — one less thing to hand-roll.

Events handled:

| `type` field                                             | → `StreamEvent`                          |
|-----------------------------------------------------------|-------------------------------------------|
| `response.output_text.delta`                               | `Delta(delta)`                             |
| `response.reasoning_summary_text.delta` / `response.reasoning_text.delta` | `ReasoningDelta(delta)` (may never appear — see liveness rule) |
| `response.completed`                                        | `Usage(usage)` (if `response.usage` present) then `Done { finish_reason: None }` |
| `response.incomplete`                                       | `Done { finish_reason: <incomplete_details.reason> }` |
| `response.failed` / `error`                                  | `Err(Protocol)` |
| anything else (`response.created`, `response.in_progress`, `response.output_item.*`, `response.content_part.*`, …) | ignored (empty event list), matching how Anthropic's parser ignores unrecognized `type`s today |

**Liveness rule (carried over from WP-02, must hold here too):** reasoning
deltas may be entirely absent for a whole stream — some targets never emit
`response.reasoning_*` events even when the model reasons. `Done` is signalled
exclusively by `response.completed` / `response.incomplete` (or `finish` at
EOF, see below), never inferred from reasoning traffic.

**EOF-without-completed:** if the transport closes before a terminal event
arrives, `SseDecoder::finish` already synthesizes a `Done { finish_reason:
None }` for every protocol (see `push_done` / the `done_emitted` guard) — this
carries over unchanged; the Responses parser needs no protocol-specific EOF
handling.

**Retry semantics:** unchanged from the other two clients — `HttpClient::
post_stream` retries only while negotiating the initial response (before
`Ok(response)` is returned for a 2xx); once bytes start flowing there is no
retry. The Responses `stream()` method calls `post_stream` exactly like
`OpenAiChatClient`/`AnthropicMessagesClient` do.

## CLI `--stream`

- Flag lives on `DispatchArgs` only (`broadcast` is out of scope — the
  assignment scopes streaming to "single-target").
- Applies **only** when the resolved chain has exactly one `BoundTarget` whose
  `transport == AdapterTransport::DirectHttp` **and** the task names no
  `--session`. Any other case (a harness target, a multi-element failover
  chain, or `--session` set) prints a one-line notice to stderr and falls
  through to the existing non-streaming `Dispatcher::dispatch` path unchanged
  — same exit codes, same ledger behavior. `--session` specifically falls back
  rather than being silently half-honored: the streaming path has no
  transcript replay or session-store update (see non-goals), so tagging a
  `RunRecord.session_id` while never touching the session store would be
  inconsistent with what `--session` means everywhere else in the CLI.
- Also falls back the same way if the protocol client's `stream()` call itself
  returns `ErrorKind::Unsupported` (defensive: covers a future direct-HTTP
  protocol client that hasn't grown streaming yet).
- On the streaming path: assemble the `ChatClient` via the same
  `Protocol → concrete client` mapping `AssemblerFactory::make` uses (no
  session continuity — single-target streaming carries no `--session` support
  in this work package, matching the "single-target DirectHttp" scope), drive
  `stream()`, print each `Delta` to stdout **with a flush per delta** so a
  human watching the terminal sees text arrive live, accumulate `Usage` and
  the full answer text, then hand-build one `RunRecord` (single `Attempt`,
  correct `status`/`usage`/`task_digest`) and append it through the same
  `Ledger` the non-streaming path uses — so `vyane history` sees streaming and
  non-streaming runs identically.
- A stream error mid-flight still yields exactly one `RunRecord` (status
  `Error`, one `Attempt::Err`) — no silent CLI abort without a ledger write.

## Known seam (see `docs/plan/feedback-wp09.md`)

`Dispatcher::dispatch` has no streaming entry point — it only exposes
`complete()`-shaped execution through `Executor::Chat(Arc<dyn ChatClient>)
.complete(..)`. Building the `RunRecord` for the streaming path therefore
duplicates a slice of `vyane-kernel::dispatch`'s record-assembly logic
(digest, attempt shape, status mapping) directly in `vyane-cli`, instead of
going through the kernel. This is a deliberate, documented shortcut scoped to
this work package (`vyane-kernel` is frozen for WP-09) — not a suggestion that
the CLI should own record assembly long-term. See the feedback file for the
proposed kernel-side evolution (a `Dispatcher::dispatch_stream` returning a
`Stream` of deltas alongside the same eventual `RunRecord`).

## Frozen interfaces touched

- Implements (no signature change): `#[async_trait] ChatClient::stream` for
  `OpenAiResponsesClient` (was the default `Unsupported` impl from
  `vyane-core::traits`).
- Consumes unchanged: `StreamEvent::{Delta, ReasoningDelta, Usage, Done}`,
  `RunRecord`/`Attempt`/`AttemptOutcome`, `Ledger::append`, `BoundTarget`,
  `AdapterTransport`.

## Acceptance (mechanically checkable)

All against `wiremock` — no live network:

- Responses SSE split across transport chunks reassembles into ordered
  `Delta`s (mirrors the existing OpenAI Chat split-frame test).
- A stream with zero reasoning events still completes cleanly (liveness rule
  — never require reasoning traffic to reach `Done`).
- `response.completed` carrying `usage` yields `Usage` then `Done` with the
  right finish semantics.
- A malformed event payload maps to `ErrorKind::Protocol` (never a panic).
- EOF with no terminal event synthesizes `Done { finish_reason: None }`,
  matching the Chat/Anthropic clients' EOF behavior.
- `vyane dispatch --stream` against a single direct-HTTP target: deltas appear
  on stdout as they arrive, and the ledger ends up with exactly one
  `RunRecord` with `status: success` and the accumulated usage/output.
- `vyane dispatch --stream` against a harness (`CliWrap`) target: falls back
  to non-streaming, prints the one-line stderr notice, ledger still gets
  exactly one record, exit codes unchanged from the non-streaming path.

## Non-goals

- No `broadcast --stream` (single-target only, per assignment).
- No `--session` continuity on the streaming CLI path.
- No kernel changes — `vyane-kernel` stays frozen; the CLI-side shortcut above
  is temporary and flagged in the feedback file.
- No new wire dependency for named-event SSE parsing; the existing
  byte-buffer decoder is reused unchanged.

## Notes

- Keep the new SSE parser functions private to `vyane-protocol::sse`, same as
  the existing two.
- Do not log stream payloads at info level (prompt/answer text may be
  sensitive) — same rule as WP-02.
