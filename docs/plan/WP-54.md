# WP-54 — Typed workflow execution plan v1

Status: integrated into the current clean public baseline.

## Outcome

`WorkflowPlan` schema v1 is the common execution payload for workflow compile,
prepare, run, and same-journal resume. It materializes the declarative DAG into
a strict, bounded representation with stable step order, typed single/fan-out
targets, route hints, lossless durations, requested capability summary, source
claim, and canonical plan checksum.

The plan deliberately omits the source file path, so an already collected plan
does not depend on the original filesystem layout. It can still contain prompts,
system text, target selectors, and absolute workdirs. It is therefore an
execution-sensitive payload, not a safe public status or API projection.

## Integrity and continuation

- `plan_sha256` binds canonical schema-v1 content and detects corruption or
  drift. It is not a signature, authentication mechanism, provenance proof, or
  execution authority.
- New journals store the plan checksum. Plan-only prepare/run/resume requires
  the exact checksum before target resolution or dispatch.
- A legacy journal without the checksum is rejected by plan-only APIs. Only the
  source-bearing `Workflow` compatibility path may migrate it, after validating
  the exact source hash.
- `source_sha256` is a collected-source or derived-programmatic claim. It is
  distinct from the plan checksum.

## Bounds and validation

The wire payload is capped before deserialization. Schema validation also caps
steps, dependency edges, per-step and global targets, route values, and total
string content. Graph validation rejects missing dependencies and cycles. The
template-ancestor closure uses topological dynamic programming and a global
relation budget, so dense valid-looking DAGs fail before resolver work rather
than causing recursive amplification.

The capability manifest is derived from the plan and summarizes requested
maximum sandbox, explicit workdir use, fan-out, route hints, and maximum timeout.
It is descriptive pre-resolution evidence only. Target resolution, capability
admission, and runtime authorization remain mandatory.

## Compatibility and limits

The existing TOML frontend retains its previous unknown-field compatibility;
the execution-plan wire is separately strict. Programmatic workflows without a
precomputed source hash receive a deterministic derived source digest. Existing
`max_concurrency = 0` input still normalizes to one when compiling, while the v1
wire uses a fixed `u32` representation.

Schema v1 supports the current declarative DAG. It does not claim support for:

- dynamic control flow or nested workflows;
- a shared workflow budget;
- an original-Vyane compatibility frontend;
- replay/fork into a new run identity;
- CLI, REST, or MCP transport of raw plan payloads;
- authenticated provenance or authority delegation.

Duplicate entries in a fan-out target list retain current execution behavior,
but their long-term compatibility semantics are not yet frozen. The bounded
32 MiB JSON envelope can still allocate container metadata during serde parsing;
the post-parse structural and global budgets bound accepted plans.

## Verification

Acceptance covers a three-finder fan-out followed by one synthesizer, canonical
JSON/digest stability, split prepare/run, exact-digest resume, tamper rejection,
legacy migration boundaries, sub-second and very large durations, programmatic
source digests, wire and aggregate budgets, and dense dependency rejection.

The delivered baseline passed workspace formatting, warnings-as-errors Clippy,
the full workspace test suite in serial mode, and Rust 1.85 locked workspace
checking.
