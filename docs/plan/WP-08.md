## SPEC — Detached background runs (docs/plan/WP-08.md)

Goal: long dispatches without a live terminal, and task management for them. No long-lived daemon — the worker is a re-exec of the vyane binary.

- "vyane dispatch ... --detach": parent validates config + resolves the target chain FIRST (config errors exit 2 before anything spawns), allocates run id (uuidv7), serializes a job spec (task, target string, workdir, sandbox, timeout, labels, session) to $VYANE_DATA_DIR/tasks/<id>/job.json, then spawns the worker: the same executable re-exec'd with a hidden subcommand "__worker <id>" (hidden from help), in its own process group (mirror the setsid approach — vyane-cli may add the same crate-local narrowly-scoped unsafe lints exception vyane-harness uses, with SAFETY comments), stdout+stderr redirected to $VYANE_DATA_DIR/tasks/<id>/task.log, then prints the id and exits 0 without waiting.
- Worker: writes status.json {schema:1, run_id, pid, pgid, state:"running", started_at, target, workdir} (atomic tmp+rename), performs the normal dispatch path (kernel + ledger as usual), then rewrites status.json with state success|error|timeout|cancelled, finished_at, answer in output.txt beside it, ledger_run_id. On panic/kill the file simply stays "running" — see orphan detection.
- "vyane task list [--json]": all task dirs, most-recent-first: id, state, target, started_at, duration. Orphan detection: state=="running" but pid dead (kill(pid,0) fails) → display state "died" (do not rewrite the file — read-side interpretation only).
- "vyane task status <id> [--json]": full status + last 10 log lines; --output prints output.txt.
- "vyane task cancel <id>": SIGTERM to the process GROUP, 5s grace, then SIGKILL group; then wait up to 2s for the worker's final status write; if still "running", report "kill delivered; worker did not finalize" and exit 1, else exit 0.
- Worker installs a SIGTERM handler that cancels the CancellationToken so the kernel path finalizes (RunRecord lands, status.json says cancelled).

Acceptance (assert_cmd, wiremock, no real CLIs):
- --detach returns fast (< a couple seconds) with an id while a slow wiremock target (delayed response) is still running; poll task status → running → success; output.txt matches the answer; ledger contains the run.
- Config error with --detach exits 2 and creates no task dir.
- cancel: long-delay wiremock + --detach → cancel → status.json cancelled AND process group dead; ledger has the cancelled RunRecord.
- Orphan: hand-craft a task dir with state running + a dead pid → task list shows died; task status exits nonzero with a clear message.
- task list ordering + --json parse.
