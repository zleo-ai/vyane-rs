# WP-56 — crates.io release authorization gate

Status: delivered in the current clean public integration baseline.

## Outcome

Crates.io publication no longer starts from a tag push. The only repository
workflow entry is an explicit `workflow_dispatch` from `main` with one required
`release_tag` input. A tag and a registry token remain release data and a
credential; neither is publication authority.

The token-free preflight job validates the exact source, runs all ordinary
checks, and verifies the full package set. Only after it succeeds does the
publish job target the GitHub environment named exactly `crates-io`. Before
checking out or executing repository code, that job's first runner step queries
the public deployment-environment API and fails closed unless that environment has
exactly one `required_reviewers` protection rule, at least one reviewer, and
`prevent_self_review: true`. This catches GitHub's otherwise dangerous behavior
of automatically creating a referenced but missing environment without
protection.

GitHub evaluates environment protection before assigning the publish runner.
The explicit first-step check is therefore a configuration-integrity assertion
after the approval boundary, not a replacement for GitHub's approval enforcement.

## Exact source identity

The workflow checks out the immutable dispatch SHA with full history and tags,
then `.github/scripts/release-gate.py` requires all of the following:

- event is exactly `workflow_dispatch`;
- dispatch ref is exactly `refs/heads/main`;
- `release_tag` has the exact `vMAJOR.MINOR.PATCH` form;
- dispatch SHA is a full lowercase commit SHA;
- checked-out `HEAD`, fetched `origin/main`, dispatch SHA, and the peeled tag
  commit are byte-for-byte identical;
- the checkout has no tracked or untracked changes.

The input reaches shell commands only through environment variables and quoted
arguments. It is never interpolated directly into a `run` program, so a hostile
manual input cannot become shell syntax before validation.

## Credential boundary

After approval, the publish job repeats the environment and exact-source checks
as its final token-free step. `CARGO_REGISTRY_TOKEN` is referenced only in the
following publish step. It is not a job-level environment variable and is
absent from source verification, Rust toolchain setup, fmt, clippy, tests,
authorization self-tests, and package preflight. Repository administration
must store it as a `crates-io` environment secret so GitHub withholds it until
the protected deployment is approved. It must not define a repository or
organization secret with the same name because the workflow cannot observe
which secret scope won GitHub's precedence rules.
The token-bearing `cargo publish` uses `--no-verify`: both jobs already built
and verified the exact archives without the credential, so the upload path does
not rerun dependency build scripts or other package code with the token in its
environment.

## Hermetic acceptance

`.github/scripts/test-release-gate.sh` uses temporary JSON fixtures and a local
bare Git repository. It performs no GitHub or crates.io writes and proves:

- a protected environment with a non-empty reviewer list and self-review
  prevention is accepted;
- missing or malformed reviewers, duplicate reviewer rules, enabled self-review,
  wrong-shaped responses, a wrong environment name, and malformed JSON are rejected;
- lightweight and annotated exact current-main tags dispatched from `main` are accepted;
- a non-manual event, non-main ref, malformed/missing/stale tag, invalid SHA,
  mismatched HEAD or remote main, and tracked or untracked dirt are rejected.

The existing WP-37/WP-39 package-set, ordering, local archive, checksum,
idempotency, retry, and registry-identity contracts remain unchanged and still
run before the token-bearing step.

## Operational prerequisite

This repository cannot safely configure or approve its own external GitHub
environment. Before any real release, a repository administrator must create
`crates-io`, configure required reviewers, enable **Prevent self-review**, and
place the registry credential in that environment. No environment, tag, secret,
approval, GitHub Release, or crates.io artifact was created while delivering
this work package.
