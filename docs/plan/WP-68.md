# WP-68 · Bounded goal acceptance verifier

Status: in progress

## Goal

Turn persisted goal acceptance descriptors into an executable, model-free local
verification surface without making verification itself a completion or pursuit
authority.

## Delivered slice

- `vyane-goal::AcceptanceVerifier` supports already-satisfied criteria,
  `manual-confirm`, `custom`, and `test-passes` descriptors.
- Executable criteria require an explicit `cmd:` prefix, use a canonical
  caller-selected workdir, scrub the inherited environment down to `PATH`,
  cap runtime at 300 seconds, retain only bounded stdout/stderr tails, and use
  a dedicated Unix process group so timeout cleanup includes descendants.
- `vyane goal verify` emits a stable verification result and persists only
  successful criteria through the existing worker/lease-fenced
  `satisfy_criterion` mutation. It never auto-completes or auto-pursues a goal.
- Manual and unsupported criteria remain `manual_required`/`inconclusive`; no
  network, GitHub, board, quota, or approval adapter is silently invoked.

## Non-goals

- No automatic pursuit, quota handoff, approval takeover, PR/board/network
  verifier, or authenticated goal service/API is claimed here.
- `cmd:` uses explicit whitespace-separated argv; shell interpretation is only
  possible when the caller explicitly chooses a shell executable in argv.

## Verification gate

- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`
- goal verifier unit and CLI acceptance tests

The parity row CON-07 remains `partial` until persisted verification evidence,
approval/continuity handoff, and bounded pursuit are separately designed.
