# WP-53 — Crash-consistent AgentRun completion handback

## Goal

Close the local crash window between an in-process AgentRun reaching a
terminal lifecycle state and its result becoming visible in an owner-scoped
domain sink. Result bodies remain outside `vyane-agent`; the AgentRun database
stores only a frozen, body-free completion descriptor.

## Delivered contract

- `vyane-agent` schema v2 records one immutable completion descriptor per
  `(owner, run, worker, generation)`. Preparation revokes ordinary execution
  effects; live or lease-loss recovery may commit only the exact permit and
  descriptor. Every other terminal path abandons it.
- Completion preparation, commit, and abandonment produce ordered body-free
  outbox events. Schema v3 adds durable projector dispositions so a missing or
  unavailable sink can be retried without starving later events, while corrupt
  or conflicting events are quarantined without deleting source truth.
- `vyane-message` schema v2 supports hidden `staged` publications. Ordinary
  reads, claims, cursors, events, and idempotency lookup cannot observe staged
  content. Exact publish or discard is owner- and digest-bound and idempotent.
- `MessageAgentCompletionSink` binds the AgentRun descriptor to the complete
  canonical message request digest and a reconstructable publication key.
  `MessageComponents::stage_completion` revalidates the completion permit and
  exact message descriptor in the same blocking task immediately before the
  message transaction.
- Lease-loss recovery inspects an injected sink only after validating the
  complete owner/run/worker/generation/prepared binding. Exact staged truth may
  become `Succeeded`; absence follows the ordinary interruption path;
  conflict or unavailable truth preserves the recovery ticket.
- `AgentCompletionPublisher` consumes committed/abandoned completion events.
  It publishes or discards idempotently before acknowledging the source event,
  so a crash between sink mutation and acknowledgement safely replays the same
  transition. Its defer delay is derived from the store-owned clock.
- `ResidentInProcessAgentSupervisor` runs execution, recovery, and completion
  publication as three isolated bounded loops. Host cancellation stops new
  cycles and drains work already in progress; it does not enqueue resume.

## Safety invariants

- No result body, prompt, credential, endpoint, path, or bearer enters an
  AgentRun row, completion event, disposition reason, or supervisor report.
- Completion authority is non-cloneable and non-serializable. Sink interfaces
  receive no controller ticket, execution permit, or raw AgentStore.
- Timeout and cancellation recovery never upgrade a staged completion to
  success. Only exact lease-loss recovery after affirmative controller loss may
  do so.
- A dropped async waiter cannot cancel an already-running blocking settlement;
  durable settlement and exact-controller tombstone release occur in the same
  non-cancellable blocking task.
- Custom stores and sinks fail closed on malformed identity, unstable metadata,
  unsupported durable dispositions, timeout, panic, or uncertain truth.

## Verification

Contract and race tests cover live commit, lease-loss recovered commit,
abandonment, owner/generation drift, concurrent cancel versus completion,
stage-future drop, recovery-future drop, sink panic, sink absence/conflict,
store-clock drift, deferred/quarantined events, cursor visibility, and replay
after sink success before outbox acknowledgement.

## Remaining parity work

This work delivers a reusable local in-process completion/handback contract. It
does not provide a concrete product operation, Process/Remote controller,
session-aware native resume, public execution API, authenticated production
host assembly, A2A/Channels adapter, live pause/resume, or automatic payload
replay. It therefore advances partial AgentRun and resident-host parity without
establishing whole-system parity.
