# WP-48 — Paired in-process AgentRun backend

Status: delivered

Integrated into the current clean public baseline, including
recovery-retirement and MSRV hardening.

## Purpose

Provide the first concrete pairing for WP-47 execution and WP-45 recovery
without introducing a resident runtime, another queue, or an ambiguous process
identity. The pairing is limited to structured effects whose ownership and
lifetime cannot escape one process-local backend instance.

## Delivered contract

`InProcessAgentComponents` binds one `InProcessAgentOperation` and one
`AgentStore` into a backend for one owner. The execution
driver and recovery driver returned by the components share that exact backend,
store, and owner; callers cannot independently assemble mismatched halves.
Only one live backend per owner is admitted process-wide, even if a competing
assembly supplies a different store pointer or operation.

The backend implements both `AgentRunExecutor` and `AgentControllerAdapter` for
`ControllerKind::InProcess`. Its registry records exact controller id and
fingerprint pairs. Recovery observes an exact live controller, signals its
cancellation token, and waits through `Notify` for exit; a fingerprint mismatch
never signals another controller. Exact absent-controller tombstones close the
race with late registration. Tombstones are bounded at 4096 and id reuse or
uncertain-confirmation capacity exhaustion fails closed. A successful durable
gone confirmation calls the same adapter with the exact controller reference
and reclaims its tombstone; failed or uncertain settlement never does. A
registration after that reclamation must pass a second durable permit
validation before operation code can run.

An operation resolves its private input only after durable execution reaches
`Running`. It receives a lifetime-bound, non-`Clone`, non-serializable effect
authority. Immediately before every model, tool, or other external effect, the
operation must obtain and consume a fresh proof backed by complete durable
permit revalidation. Dropping the operation future must synchronously stop
ownership of its effects; detached or unowned work violates the port contract.

## Verification

Hermetic tests cover paired-driver shared state, process-wide
one-live-backend-per-owner exclusion even across different store pointers,
exact fingerprint isolation, cancellation observation, exit waiting,
late-registration tombstones, bounded retirement, confirmed reclamation beyond
the bound, reuse rejection, pre-operation and per-effect permit revalidation,
revocation, and uncertain execution/recovery races.

## Non-goals and remaining parity work

WP-48 supplies no `Process` or `Remote` backend, resident supervisor, message
input/correlation/handback, service/CLI/daemon production assembly, session-aware
resume, live pause/resume, or automatic replay. `InProcessAgentOperation` is a
trusted port, not a concrete product operation. The components therefore make
the one-shot execution/recovery contracts concrete only for one narrow
process-local controller domain.
