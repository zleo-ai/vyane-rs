# WP-73 · Visible quota continuity state

Status: in progress

## Goal

Persist a typed goal continuity policy and idempotently turn normalized quota
facts into a visible handoff plan. This slice records what may happen next; it
does not authorize or start takeover work.

## Contract

- A policy keeps provider, protocol, harness and model as separate target axes.
- Only `in_progress` goals with `quota_handoff` policy are eligible.
- A direct goal id takes precedence. Otherwise the quota target must match the
  policy's primary provider, harness and optional model.
- The same quota event id is applied once. The latest fifty applied ids remain
  in the bounded snapshot so replay stays idempotent across restart.
- A configured takeover produces a `takeover_ready` three-step plan: takeover,
  review and primary resume after quota reset. A missing takeover produces the
  explicit `blocked_no_takeover` / `manual_decision` state.
- State update and its body-free progress event share one SQLite transaction.
  Raw provider errors and prompts are never persisted.

## Explicit limits

- No quota-ledger reader, polling runner, automatic dispatch, approval inbox,
  step transitions, review evidence, handback execution or authenticated API.
- Target resolution remains outside this crate. A plan target is declarative,
  not execution authority.
- This does not change goal lifecycle status or bypass a worker lease.

## Verification gate

- Contract tests for visible plan shape, replay idempotency, target/status
  filtering, manual blocker state and schema migration.
- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`

CON-07 remains `partial`: this closes policy metadata and visible quota state,
while approval, takeover execution, review/handback and authenticated service
exposure remain separate work.
