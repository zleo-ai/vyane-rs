# WP-49 — Principal-derived owner service scope, phase A

Status: delivered

Integrated into the current clean public baseline, including
authentication-boundary and scoped-streaming hardening.

## Purpose

Remove the fixed-owner assumption from the shared service boundary while
keeping authentication, protocol wiring, task-control schema migration, and
administrative authority as explicit later phases.

## Delivered contract

`OwnerContextFactory` freezes an application-owned `PrincipalAuthenticator`
and `PrincipalOwnerResolver`. A protocol handler can submit credential bytes
to that factory, but it cannot publicly construct `AuthenticatedPrincipal` by
relabeling a request field. The authenticator's verified subject is resolved to
a canonical durable owner, from which `OwnerContext` is minted. Authenticated
resolution rejects the reserved `local` namespace; only the explicit
single-user compatibility constructor can mint it. Principal, context, and the
closed `OwnerContextError` taxonomy are non-serializable; principal/owner
values and authentication/resolver errors are redacted from diagnostics.

`VyaneService::scope` freezes one `OwnerContext` into an
`OwnerScopedService`. The facade binds the same owner to its dispatcher and to
run history, session listing/inspection, revision-checked native-session reset,
and single-target streaming dispatch. The REST SSE path now uses this scoped
seam even while its current bearer still selects the explicit local
compatibility context. Foreign sessions remain absent-shaped. Static
configuration and route inspection do not become cross-owner authority.

Existing CLI and MCP callers retain an explicit trusted single-user `local`
compatibility path. That compatibility surface is intentional and does not
derive owner authority from untrusted input.

## Verification

Hermetic tests cover authentication/resolution failure, reserved-local
rejection, redacted closed errors, non-serialization, two owners using one
service without run/session cross-read or cross-reset, owner-bound
dispatch/broadcast, and preservation of the legacy local compatibility
behavior. Streaming uses the same frozen dispatcher seam.

## Non-goals and remaining parity work

The REST bearer boundary does not yet authenticate distinct principals through
`OwnerContextFactory`; its service calls are all frozen into one trusted
single-user `local` scope at router assembly. Durable task truth is
owner-qualified after WP-50, but the
REST/CLI/daemon control planes still intentionally select only `local`.
Workflow, message, AgentRun, event subscription and mutation surfaces are not
all wired through this facade.

Phase A therefore does not close GOV-01 or provide hostile same-UID/multi-user
isolation. Cross-owner administration will require a separate typed capability;
an owner string, nullable owner, or ordinary `OwnerContext` must never imply
administrator authority.
