# WP-81 · Bounded typed goal observation ingress

## Outcome

Add an opt-in, owner-frozen mutation port that converts finite typed watcher
observations into the existing continuity signal transaction. A one-shot poller
fetches a bounded watcher set concurrently, then ingests valid batches in a
deterministic order. This package observes facts only: it has no approval,
dispatch, retry, completion or periodic-run authority.

## Ingress contract

- `GoalObservationIngress` freezes a durable owner supplied by an existing
  `OwnerContext`. No observation or watcher can select or override that owner.
- A trusted assembler must bind one bounded, non-secret source identity before
  it receives a `GoalObservationSink`. Observations contain no owner, source,
  endpoint, credential or executable command.
- The closed observation kinds are `quota_reset`, `review_checks_passed` and
  `review_checks_failed`. Each carries the exact current quota event and primary
  provider/harness/model boundary. Review observations additionally carry the
  existing repository, pull request, stable observation ID and monotonic
  sequence evidence.
- The sink delegates to the one existing `GoalStore::record_continuity_signal`
  transaction. Exact replay stays idempotent; stale review evidence cannot
  reopen a gate; mismatched boundaries and invalid observations fail closed.
- Foreign and absent goal IDs remain indistinguishable to the owner-frozen
  sink. Receipts expose only a validated goal ID, closed signal kind and a closed
  `recorded`/`unchanged`/`absent`/`rejected`/`unavailable` status. They exclude
  owner, source, target, repository, persisted state, event detail and raw
  errors. An invalid goal ID is rejected without echoing it.

## Watcher seam

- `GoalObservationWatcher` has one stable identity and returns a finite typed
  batch. Concrete adapters own their endpoint and credential configuration;
  neither may be supplied by an observation.
- Construction rejects invalid or duplicate watcher identities and bounds
  watcher count, fetch concurrency, timeout and per-watcher batch size.
- `poll_once` isolates watcher error/timeout, sorts completed fetches by watcher
  ID and applies each accepted batch in watcher-returned order. An oversized
  batch is rejected before any item from that batch is written.
- Watcher failures cross the boundary only as a closed redacted error code.
  The crate supplies no concrete network/platform watcher or credential loader.

## Verification gate

- Tests prove owner and source freezing, exact replay idempotency, foreign-scope
  isolation and no-write rejection for absent/mismatched facts.
- Typed quota and review-check observations exercise the existing durable
  sequence and boundary fences.
- Runner tests prove duplicate/configuration rejection, concurrency/timeout
  isolation, stable report order and all-or-nothing oversized-batch rejection.
- Mutation tests must fail when owner freezing, source binding or the
  no-partial-write batch fence is removed.
- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`

## Explicit exclusions

- no HTTP, MCP or CLI ingestion endpoint;
- no concrete GitHub, quota-provider or network watcher;
- no credential loading, persistence, scheduling or periodic loop;
- no approval queue/decision/consumption, execution, retry or repair runner;
- no goal completion or acceptance mutation;
- no runtime-native session resume;
- no reuse of the WP-80 read bearer as mutation authority.

## Dependency boundary

WP-82 may separately authenticate a bounded runner that reads the WP-80
projection and invokes explicitly configured queue/decision/execution ports. It
may use this observation port only under its own mutation authority. Manual
decision and one-shot approval/dispatch fences remain mandatory.

CON-07 and OBS-06 remain `partial` after this package.
