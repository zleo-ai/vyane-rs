# WP-63 тАФ Durable AgentRun execution-backend routing

## Outcome

Every new AgentRun freezes one closed execution-backend discriminator before
queue insertion. SQLite schema v4 persists `CliHarnessProcess`,
`NativeInProcess`, or `Remote`; schema-v3 rows migrate to
`LegacyUnassigned` because their original execution implementation cannot be
proven from durable truth. Unassigned rows are never claimable or resumable.

This package is routing infrastructure. It does not add a native or remote
runtime and does not make multiple complete resident supervisors safe to run
side by side.

## Claim and controller contract

`AgentStore::claim_due` requires an owner and an exact backend. Candidate
selection filters by both, while the per-worker earlier-queued check remains
backend-agnostic. A lane therefore cannot skip an earlier due run for the same
worker merely because that run belongs to another backend. Runs for independent
workers remain available to their matching lanes.

The execution driver freezes an executor's controller kind once, maps it to the
closed backend, and passes that backend to the store claim. Before controller
admission, reservation, or execution, it rejects a custom-store response whose
record does not carry the exact requested backend. Store start also requires
the live controller kind to match the frozen backend. Resume inherits the
source backend without a caller override.

## Migration and failure behavior

Schema v4 adds the bounded backend column and a backend-aware due index. The
v3 manifest and data are audited before migration. Existing rows receive the
unassigned sentinel rather than a guessed Process value. New submissions and
claim calls cannot select that sentinel, and resume from an unassigned source
fails closed. Tests cover migration, claim denial, resume denial, backend
filtering, cross-backend FIFO, controller mismatch, and a malicious custom
store returning the wrong backend.

## Follow-up boundary

A production multi-backend host still needs multiple execution lanes around
one shared recovery union and one completion projector. Running one full
`ResidentAgentSupervisor` per backend would duplicate unfiltered recovery and
completion polling, allowing the hosts to contend for the same durable work.
Native production assembly, Remote execution, and that union host remain
separate work packages.
