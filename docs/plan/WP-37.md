# WP-37 — crates.io release integrity

Status: superseded by [WP-56](WP-56.md) for release authorization and workflow
entry gating. Its package ordering, artifact identity, retry, and registry
verification contracts remain in force.

## Goal

Make the original tag-triggered release fail closed: all workspace packages must be
packageable, the tag must match their shared version, and no partial or failed
upload may be reported as a successful GitHub Actions run.

## Scope

- `.github/workflows/publish.yml` runs the ordinary format, clippy, and test
  gates with the lockfile enforced.
- `.github/scripts/publish-workspace.sh preflight` validates the release tag,
  the exact workspace package set, one shared version, and dependency order,
  then runs `cargo package --workspace --locked`. Cargo's temporary local
  registry lets this build and verify every packaged crate before any of them
  exists on crates.io.
- Preflight removes Cargo's generated `target/package` staging directory and
  uses a fresh, dedicated packaging `CARGO_HOME`. Cargo caches unpacked local
  registry packages outside `target/package`; isolating both locations prevents
  a rerun at the same workspace version from verifying dependents against stale
  archives.
- `.github/scripts/publish-workspace.sh publish` publishes exactly one crate per
  command in dependency order. It waits for that exact version to be visible on
  crates.io before advancing.
- Rerunning a partially completed release is idempotent: an exact crate/version
  is skipped only when its crates.io metadata identifies the expected repository,
  it is not yanked, and its checksum matches the local `.crate` generated by
  preflight. Registry request failures, foreign/yanked artifacts, unexpected HTTP responses,
  and upload failures still fail after bounded retries.
- The workflow uses one non-cancelling concurrency group so two release tags
  cannot race in the crates.io namespace.
- Every third-party GitHub Action is pinned to a full commit SHA. Comments retain
  the human-readable upstream tag/branch that was resolved from the action's
  official repository.

10. `vyane-workflow`
11. `vyane-service`
12. `vyane-mcp`
13. `vyane-cli`

The helper verifies this list against `cargo metadata`, including every
intra-workspace dependency edge. A new crate or dependency-order change blocks
the release until the list is intentionally updated.

## Failure and retry semantics

- `vX.Y.Z` must equal the version of every workspace package exactly.
- Preflight checks all thirteen package artifacts; checking only `vyane-core` is
  insufficient because package manifests differ after path dependencies are
  rewritten for crates.io.
- `cargo publish` is retried a bounded number of times to tolerate sparse-index
  propagation between dependent packages.
- A failed upload command is considered successful only if the crates.io API
  returns the expected crate/version, `yanked` is exactly `false`, repository
  identity matches, and the published checksum equals the local package archive.
  This covers the narrow case where the upload landed but the client lost its
  response without trusting an unrelated artifact.
- Missing or malformed identity fields fail closed. The final step repeats the
  full identity/checksum audit for all thirteen artifacts.
- Preflight runs the real verifier against synthetic valid, yanked,
  foreign-repository, wrong-checksum, wrong-name, wrong-version, and
  missing-field responses. Publishing is unreachable if any fail-closed branch
  regresses.

Retry counts and delays can be tuned with `PUBLISH_MAX_ATTEMPTS`,
`PUBLISH_RETRY_DELAY_SECONDS`, `PROPAGATION_MAX_ATTEMPTS`, and
`PROPAGATION_RETRY_DELAY_SECONDS`; all must be positive integers.

## Acceptance

- A mismatched tag exits before packaging or publishing.
- Missing, extra, duplicate, differently-versioned, or topologically
  misordered workspace packages fail validation.
- `cargo package --workspace --locked` succeeds for all thirteen crates.
- Package staging and its Cargo registry cache are rebuilt from empty state, so
  same-version residue cannot satisfy dependency verification or checksum
  validation.
- All thirteen generated `.crate` archives exist and have valid SHA-256 digests.
- Each upload is a separate `cargo publish --package <crate>` command; no shell
  construct converts an unverified failure into success.
- A rerun skips only exact, non-yanked, repository-matching versions whose
  crates.io checksum matches the local `.crate`, then continues from the first
  missing package.
- A yanked, foreign-repository, malformed, or checksum-mismatched HTTP 200
  response is a hard failure in the initial check, ambiguous-success check, and
  final audit.
- The synthetic identity-verifier self-check accepts the one valid fixture and
  rejects all six invalid fixture classes before publishing.
- A permanent upload or crates.io API failure makes the workflow fail.
- A successful workflow ends only after all thirteen exact artifacts are verified.

## Non-goals

- Creating the crates.io token or assigning crate owners.
- Creating a GitHub Release, changelog, or binary artifacts.
- Automatically changing workspace versions or creating tags.
