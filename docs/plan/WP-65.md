# WP-65 — Dark fresh-sessionless native AgentRun operation

## Status

Implemented as a dark, unregistered composition slice. The repository already contains the lower-level native authority,
authorized OpenAI Chat client, bounded `NativeTurnDriver`, in-process AgentRun
backend, durable backend routing, message-completion handback, and multi-lane
host seams. This package proves their exact composition as one dark operation;
it does not make a production-runtime claim.

## Intended outcome

Add a private, create-only native input spool that freezes one bounded prompt
and the exact secret-free target and policy material needed by a native
AgentRun. Loading an item must bind its owner, run, worker, prompt digest, and
policy digest without exposing those values in its storage path or debug
output. Inputs are fresh and sessionless only.

The native operation resolves the frozen direct OpenAI Chat target through the
authorized client factory, derives the exact `NativeExecutionScope`, binds it
to the active in-process effect authority, and gives the resulting authority
to a tool-free `NativeTurnDriver`. The driver may perform only authorized model
sends; no tool implementation is installed by this package.

One hermetic end-to-end test must enqueue an exact
`NativeInProcess` AgentRun, consume its private spool item through the real
in-process operation, complete a bounded OpenAI Chat turn against a fake local
endpoint, settle the durable run, and publish the exact message completion.
The test must also prove that prompt or policy drift fails closed before a wire
effect and that public status/output surfaces do not expose the private input.

## Acceptance boundary

The implementation is complete only when the code and tests prove all of the
following:

- the spool is bounded, owner-private, create-only, identity-bound, and
  fresh/sessionless;
- the operation accepts only the exact `NativeInProcess` backend and exact
  frozen direct OpenAI Chat target;
- model construction goes through the authorized native client factory;
- every physical model attempt remains guarded by the exact live native
  authority;
- the `NativeTurnDriver` has an empty advertised and executable tool set;
- success is handed back through the existing durable message-completion path;
- cancellation, timeout, malformed input, target drift, policy drift, and an
  unavailable completion sink fail closed without inventing replay or resume.

## Explicit non-goals

WP-65 does not register a native target with the daemon, CLI, REST, or MCP API,
and it does not add a native lane to the production host. It adds no trusted
built-in tools, OS sandbox, session continuity, resume authority, checkpoint
or session commit, approval flow, failover, automatic replay, or restart
reconstruction. It is not general native-production parity.

Those capabilities require separate work packages and their own executable
acceptance evidence.

## Verification

- `cargo test -p vyane-cli native_agent` exercises the real private spool,
  in-process backend, authorized client, mock HTTP endpoint, durable settlement,
  staged completion, and publication path.
- `cargo test -p vyane-service` covers the controller lifetime and completion
  staging contracts, including cancellation while the non-cancellable blocking
  stage remains active.
- `cargo test --workspace`, workspace Clippy with warnings denied, and the
  formatting gate pass on the complete change.
- Mutation checks proved the acceptance tests are sensitive to endpoint routing
  drift, conditional pending-slot cleanup, and moving private cleanup outside
  the recovery-visible completion-stage lifetime.
- Independent review found no remaining P0, P1, or P2 findings after the
  stage/cleanup race and one-shot pending-slot leak were fixed.
