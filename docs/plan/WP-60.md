# WP-60 — Owner-scoped goal truth and lifecycle CLI

## Outcome

Vyane has a durable local goal ledger and a machine-stable lifecycle CLI. The
Rust design preserves the useful lifecycle and queue behavior while replacing a
split writable JSONL/index design with one SQLite source of truth.

## Storage contract

- `vyane-goal` owns `goals.sqlite3`.
- `goals` is the current query snapshot; `goal_events` is immutable lifecycle
  and progress history.
- Create and every mutation use an immediate transaction. Snapshot and event
  carry the same revision and commit or roll back together.
- Goal events reject update and delete at the database boundary.
- Snapshot keys, event foreign keys, reads, lists, queue selection and
  transitions all include owner. The same id may exist under two owners;
  foreign and absent records have the same observable result.
- Queue ordering is priority ascending, creation time ascending, then id.
- The schema rejects unknown future versions instead of attempting downgrade.

The lifecycle is:

```text
queued      -> in_progress | cancelled
in_progress -> completed | failed | paused | cancelled
paused      -> in_progress | cancelled
completed | failed | cancelled -> terminal
```

Progress increments the revision and appends an event without changing status.
Acceptance criteria persist as bounded `kind` and `target` descriptors with an
optional future satisfaction timestamp. Persisting a descriptor does not grant
authority to execute it.

## CLI contract

`vyane goal` provides:

- `create`, `get`, `list`, `next`
- `start`, `progress`, `pause`, `resume`
- `done`, `fail`, `cancel`

All commands support `--json`, `--owner` (with `--owner-user-id` alias), and
`--db`. Success is exit 0. Post-parse validation, missing records, illegal
transitions and store failures are exit 2 and use
`{"status":"error","error":"..."}` under `--json`; clap parse failures retain
clap's exit-2 diagnostics. The default database is the standard Vyane data directory's
`goals.sqlite3`.

`--owner` is local caller-selected storage scope, not authenticated principal
authority. The CLI must not be exposed directly as a hostile multi-user
service.

## Acceptance

- Store contract covers the full lifecycle, revision-ordered events,
  deterministic queue/list ordering, same-id cross-owner isolation, foreign-as-
  absent reads and mutations, immutable events, input bounds and reopen/newer-
  schema behavior.
- An injected event-insert failure proves the snapshot update rolls back in the
  same transaction.
- CLI acceptance covers lifecycle round-trip, persisted acceptance descriptors,
  queue auto-start, state filtering, same-id cross-owner behavior, JSON error
  exit 2, the default database path and unauthenticated-scope help text.
- Mutation proof: temporarily removing the owner predicate from exact goal reads
  makes `owner_scope_allows_same_id_and_hides_foreign_records` fail by returning
  the first owner's title for the second owner.
- Workspace fmt, clippy and tests remain green.

## Non-goals

- No acceptance verifier or command execution.
- No automatic pursuer, completion authority, quota handoff or continuity
  engine.
- No REST, MCP, daemon or authenticated principal wiring.
- No multi-device synchronization, distributed fencing, retention or archive.
- No writable JSONL mirror or second source of truth.
